03 Agent Core Prompt assembly, tool projection, and the iterative agent loop.
Agent Core
Prompt Assembly + Iterative Loop
Agent resolves provider/model, refreshes tokens, reads dynamic projection state from the active tool registry, injects only tools with inject_into_prompt=true, and appends discovery guidance for omitted tools (list_tools/search_tools, tools.<tool>). Prompt assembly is execution-mode aware: the same tool identity is projected into REPL help/docs or provider-native tool schemas depending on the active mode.

Prompt Tooling

Visible tools split into prompt-injected subset + discoverable runtime subset from the active generation.

Execution Mode

repl and native-tools use different model-output contracts, but both flow back into the same AssembledTurn host result.

Context Control

Batched context folding with soft/hard watermarks plus _history / _mem for long-running sessions and sub-agents.

Capability Guidance

Prompt helper bindings + capability prompt sections come from dynamic capability definitions.

04 REPL Mode Embedded Python execution with persistent state across turns.
Execution Contract
REPL Execution Model
The session spins up embedded Python, registers tool definitions/capabilities, and executes <repl> blocks in a persistent namespace. The model may emit prose plus at most one <repl> block per turn.

Contract Rules

<repl> blocks execute immediately when </repl> is reached. Maximum one block per turn. After execution, lash continues in a new internal turn with execution output. Variables persist across turns. print(...) output is model-visible only.

Termination

done(value) ends the turn and returns control. May only be called inside <repl>. In interactive mode, call done(...) only when the final user-facing answer is ready. In headless mode, call done(...) only after the task is fully completed.

Tool Access

All visible tools are available in the tools module (e.g. tools.read_file(...)). from tools import * is applied automatically. Use await only for long-running calls (delegate_*() and shell handle methods).

Error Recovery

Tool failures raise ToolError; execution stops at the failing line. Read the traceback, fix the cause, retry. If REPL state is corrupted, call reset_repl().

State Persistence

Python locals persist across turns within the same session. The REPL namespace can be snapshotted and restored on session resume.

Headless Constraint

In headless mode, prose-only turns are invalid. The model must always execute via <repl> and finish with done(...).

05 Native Tools Provider-native tool call execution with structured results.
Execution Contract
Native Tools Execution Model
The Rust agent loop sends tool schemas to the provider. The model responds with native tool calls when execution is needed — no Python, no <repl>. Tool results are fed back as structured content for the next turn.

Contract Rules

Use native tool calls when execution is needed. Do not emit <repl> blocks or Python code. When several tool calls are independent, emit them together in the same response. Use serial calls only when later arguments depend on earlier results.

Termination

Finish by returning a final assistant answer. No fake tool calls or placeholder arguments. Do not stop after tool execution unless the task is actually complete. In headless mode, keep calling tools until the task is done, then return a final answer.

Tool Access

The runtime exposes visible tools as native tool call schemas. Use only the tools shown in Available Tools. Prefer parallel independent calls in one turn. If batch is available, prefer it for 2+ independent calls. Never invent tool names or arguments.

Parallelism

Good candidates for one-turn parallel calls: reading several files, multiple searches, independent inspections. Do not parallelize dependent steps, ordered stateful edits, or operations whose arguments depend on prior output.

No Persistent State

Unlike REPL mode, there is no Python namespace. Each tool call is stateless. Context comes from the transcript and prior tool results only.

Provider Streaming

Native mode streams structured tool-call parts from the provider layer. Grouped multi-call turns are replayed back to providers where the API supports it.

06 Tools & Delegation Composite tool graph, dynamic providers, and sub-agent delegation.
Tools
Composite Tool Graph
Static providers are wrapped by DynamicToolProvider (in-process adapter), then projected by dynamic capability profiles. CLI can stage edits via /tools and /caps, with apply on next turn.

Core Tools

read_file, edit_file, write_file, find_replace, shell, glob, grep, ls.

Reconfigure

All-or-nothing apply with generation checks; failures roll back and block turn dispatch.

Discovery

list_tools, search_tools, and help(tool_name) let the model discover tools not injected into the prompt.

Delegation
Sub-Agent Orchestration
agent_call forks parent dynamic state per sub-agent. Sub-agents share the same filesystem/worktree but get isolated tool/capability state. Low tier sub-agents are filtered (no write/delegation tools) and always execute in native-tools; medium/high inherit the parent session's execution mode.

Tiers

Low (read-only lookup, forced native-tools), medium (general), high (full capability). Each tier has a filtered tool set.

Recall Pattern

Use a low-tier read-only sub-agent to summarize _history/_mem when prior context is needed.

Constraint

Parallel sub-agents should avoid editing the same files concurrently.

Persistence
Store, Logs, and Session State
SQLite-backed store tracks tasks/context artifacts for the session, while CLI keeps session logs and runtime state envelope that can be restored on resume.

Store

Session-specific DB under ~/.lash/sessions/*.db.

Envelope

AgentStateEnvelope: messages, iteration, token usage, task/subagent state, snapshot.

07 Capabilities Dynamic capability profiles that control which tools and prompt sections are active.
Capability System
Dynamic Capability Profiles
Each capability is a named group that bundles tools, prompt sections, and helper bindings. Capabilities can be toggled at runtime via /caps. All are enabled by default. Sub-agents inherit a fork of the parent's capability state.

Structure

Each CapabilityDefinition has an id, name, description, optional prompt section, helper bindings (REPL globals), and a list of tool names it controls.

Projection

Only tools belonging to enabled capabilities appear in the active tool registry. Prompt sections from enabled capabilities are injected into the system prompt automatically.

Capability Registry
Built-in Capabilities
Ten capability groups ship by default. Each controls a coherent set of tools and optional prompt guidance.

Core Read

read_file, glob, grep, ls, search_tools. File navigation and discovery.

Core Write

write_file, edit_file, find_replace. File mutation tools.

Shell

shell, shell_wait, shell_read, shell_write, shell_kill. Process handles.

Tasks

tasks, tasks_summary, get_task, create_task, update_task, delete_task, claim_task.

Planning

enter_plan_mode, exit_plan_mode. Interactive plan lifecycle.

Delegation

agent_call, agent_result, agent_output, agent_kill. Sub-agent orchestration.

History

search_history, history_add_turn, history_export, history_load. Persistent turn history.

Memory

search_mem, mem_set, mem_get, mem_delete, mem_export, mem_load. Key-value memory.

Skills

Web

08 Memory & History Persistent key-value memory and turn history that survive context pruning.
Memory
Persistent Key-Value Memory
_mem is persistent key-value memory across context pruning. Use it to store durable decisions, conventions, and project-specific knowledge. Available as a REPL global and via tool calls.

API

_mem.set(key, description, value=None) — store/update
_mem.get(key) — retrieve or None
_mem.delete(key) — remove
_mem.all() — list keys with descriptions
_mem.search(query, mode, regex, limit, keys) — search

Built-in Helper

search_mem(query, mode="hybrid", regex=None, limit=10, keys=None) is available as a top-level built-in in REPL mode and as a native tool in native-tools mode.

Example — Storing a convention (REPL)
<repl>
_mem.set("repo_convention", "Use snake_case for tool names")
print(_mem.all())
</repl>
Example — Searching memory (REPL)
<repl>
results = search_mem("naming convention")
for r in results:
    print(r.key, r.description)
</repl>
Example — Searching memory (Native Tools)
assistant:
tool_call: search_mem
  query: "naming convention"
  limit: 5
History
Persistent Turn History
_history tracks prior turns across context pruning. When the rolling context window drops old messages, they are preserved in history for retrieval. Sub-agents can inherit parent-agent context at spawn time.

API

_history.user_messages() — list prior user messages
_history.search(query, ...) — search pruned turns
search_history(query, mode="hybrid", regex=None, limit=10, fields=None, since_turn=None)

Sub-Agent Inheritance

_history and _mem can include parent-agent context inherited at spawn time. A low-tier sub-agent can summarize relevant history without full write access.

Example — Recalling prior context (REPL)
<repl>
# Find what the user asked about earlier
hits = search_history("database migration", limit=5)
for h in hits:
    print(h.turn, h.role, h.snippet)
</repl>
Example — Delegating history recall (REPL)
<repl>
# Use a cheap sub-agent to mine history
summary = await delegate_low(
    "Search _history for any database schema decisions and summarize them."
)
print(summary)
</repl>
Use _history and _mem only when prior-turn recall is actually needed. Skip history-mining detours unless the task explicitly depends on prior turns.
09 Skills Discoverable, loadable skill packages that extend agent behavior at runtime.
Skills System
Skill Discovery & Loading
Skills are directory-based packages containing a SKILL.md manifest with description and instructions, plus optional supporting files. They live in ~/.lash/skills/ (global) and .lash/skills/ (project-local), with project-local overriding global by name.

Structure

Each skill is a directory with a SKILL.md file. The manifest contains a description and instructions that get injected into the agent's context when loaded. Optional files alongside the manifest are accessible via read_skill_file.

Discovery

skills lists all available skills with names and descriptions. search_skills(query, ...) performs hybrid search across skill metadata. Both global and project-local skill directories are scanned.

Loading

load_skill(name) returns the full skill definition including instructions and file listing. The agent then follows the skill's instructions to complete the requested workflow.

Example — Discovering skills (REPL)
<repl>
# List all available skills
all_skills = skills()
for s in all_skills:
    print(s.name, "-", s.description)
</repl>
Example — Loading and using a skill (REPL)
<repl>
skill = load_skill("frontend-design")
print(skill.instructions)
# Agent now follows the skill's instructions
</repl>
Example — Searching skills (Native Tools)
assistant:
tool_call: search_skills
  query: "UI component library"
  limit: 5
Skill Manifest
SKILL.md Format
The manifest is a markdown file parsed for name (directory name), description (first paragraph), and instructions (full content). Additional files in the skill directory are listed and accessible.

Resolution Order

~/.lash/skills/ is scanned first (global), then .lash/skills/ (project-local). Project-local skills override global skills with the same name.

File Access

read_skill_file(skill_name, file_path) reads supporting files bundled with the skill, such as templates, schemas, or reference code.

10 Dataflow Primary execution paths from host input through the agent loop to streamed output.

Shared Turn Lifecycle

01 Host Input

Frontend emits ordered TurnInput.items + optional image blobs.

02 Normalize + Apply

Runtime validates refs and applies pending dynamic reconfigure before turn start.

03 Agent Step

Prompt assembled from a mostly static prefix plus later dynamic sections for environment, memory, and tools, using the current execution mode.

04 Mode Exec

Execution diverges here: persistent Python REPL for repl, provider-native tool loop for native-tools.

05 Fold + Output

When prompt usage reaches the hard watermark, old turns are folded back to the soft watermark with one stable archive marker; events stream back and the final answer + updated envelope return as one canonical AssembledTurn.
