lash architecture dossier

Protocol/Host

lash is a Rust coding-agent workspace with a stricter boundary than before: lash-sansio owns the protocol and turn machine, lash owns the async runtime and plugins, lash-ui owns host-level UI extensions, lash-cli is one terminal host, and lashlang provides the local REPL language. The important current rule is that core emits host-neutral events and effects; slash commands, shortcuts, queued follow-up turns, and terminal affordances live above that boundary.

What lash is

A layered coding-agent stack that can be embedded below the CLI. The core runtime manages providers, tools, plugins, sessions, and persistence; the terminal app is only one host over that substrate.

  • lash-sansio owns TurnMachine, protocol effects, session events, prompt types, and shared message/tool contracts.
  • lash owns runtime orchestration, provider transport, tool dispatch, plugin registration, child sessions, and persistence.
  • lash-ui owns UI extensions, host actions, slash-command/shortcut registration, and agent-event-to-host-effect translation.
  • lash-cli owns bootstrap, full-screen terminal rendering, interactive input, autonomous mode, and the stock host implementation.
  • lashlang owns the local workflow DSL used by REPL execution mode.

What changed architecturally

The recent clean cut introduced lash-ui so the runtime no longer needs to know about slash commands, shortcuts, or TUI-owned follow-up behavior. Plan mode is the first proof case: the core plugin still owns planning policy, but the terminal affordances now live above it.

  • SessionEvent::Prompt and PluginSurfaceEvent remain host-neutral runtime transport.
  • UiExtension now maps host commands, shortcuts, and session events into UiHostEffect values.
  • plan_exit follow-up queueing moved out of App special-casing and into the plan-mode UI extension.
  • The SQLite transcript model from the previous refactor remains intact: semantic rows are still the only durable conversation truth.

UI extension proof case

  • plan_mode.rs remains a core plugin that owns planning rules, tool gating, and plan approval.
  • lash-ui owns /plan, Shift+Tab, idle indicator sync, and plan-mode surface cleanup after plan_exit.
  • The CLI host only applies generic UiHostEffect values.

Current hotspots

  • lash/src/runtime.rs — orchestration, streaming, checkpoints, child sessions.
  • lash-cli/src/render/mod.rs + lash-cli/src/scratch_tui.rs — terminal projection, overlays, and paint/layout behavior.
  • lash-cli/src/app/mod.rs — transcript projection, turn finalization, block management.
  • lash-cli/src/interactive/mod.rs — event pump, input semantics, host command dispatch.

Deliberate split

  • The protocol core does not know about slash commands, key chords, or terminal widgets.
  • The store does not persist pre-rendered UI blocks as transcript truth.
  • lash-ui is host-facing policy, not durable runtime truth.

Data Flow

How user input, runtime events, and host effects move through the current layered design. There are now two distinct flows: canonical transcript flow and host-affordance flow.

flow

Interactive host pipeline

User drafts and host affordances meet in the CLI, but only semantic turns go through the canonical transcript path.

Single source of truth

  • transcript_entries is the canonical persisted transcript table.
  • Two keyspaces matter today: message and tool_call.
  • load_session() always rebuilds blocks from semantic transcript rows.
  • load_picker_info() also queries transcript rows directly rather than reading a UI cache.

What is not transcript truth

  • session_state stores typed runtime metadata such as prompt usage, replay manifest, plugin snapshot, dynamic tool state, and task state.
  • ui_resume_state stores only live affordances that cannot be derived from completed transcript rows.
  • UiHostEffect values such as mode-indicator updates, queued turns, and host messages are ephemeral host actions.
  • Examples: streaming tail lines, plugin panels, idle mode badges, and partial assistant text during an interrupted turn.
resume path

Load projection steps

  1. Open the session database and fetch transcript rows.
  2. Rebuild Message[] and ToolCallRecord[] from the transcript.
  3. Project those semantic records into DisplayBlock values.
  4. Overlay UiResumeState for live assistant text or plugin panels when needed.
host path

Extension effect steps

  1. Build the UiExtensions registry at CLI startup.
  2. Sync extension state via runtime external ops such as plan_mode.status.
  3. Convert user shortcuts/commands and session events into UiHostEffect values.
  4. Apply those effects locally without changing the runtime protocol or transcript schema.

Standard mode

  • The runtime passes tool schemas to the provider.
  • The model emits native tool calls directly.
  • The host executes those calls and feeds ToolResult data back into the machine.

REPL mode

  • The prompt includes rendered tool docs instead of provider tool schemas.
  • The model emits fenced lashlang programs.
  • The host executes those programs locally and streams code/output records back into the transcript.

Host-extension mode

  • Shortcuts and slash commands can be plugin-owned without changing lash-sansio.
  • Extension sync is explicit at startup and after session resets/resumes.
  • Plan mode is the first built-in example; other hosts can provide different UI adapters over the same runtime.

Checkpoint behavior

The runtime persists during turn execution via save_turn_checkpoint(), writing a durable agent snapshot and canonical transcript keyspaces. That lets interrupted turns resume without inventing a separate transcript format.

Finalization behavior

After the plugin finalize step, the runtime saves the latest semantic state again. The CLI then reconciles any live assistant overlay against the finished assistant message text and applies any queued host effects produced by lash-ui from the final event stream.

Abstractions

The contracts that keep the system modular: the machine/runtime boundary, the store boundary, and now the explicit runtime/host boundary above that.

Turn protocol

  • TurnMachine, Effect, Response, TurnMachineConfig
  • Defines the host/core boundary. No direct I/O happens inside the machine.

Runtime host

  • LashRuntime, RuntimeHostConfig, EventSink, AssembledTurn
  • Owns provider transport, tool dispatch, session lifecycle, and checkpoint coordination.

UI host layer

  • UiExtension, UiExtensions, UiHostEffect, SlashCommandSpec, ShortcutSpec
  • Lets a host own commands, shortcuts, queued turns, and mode indicators without changing the runtime protocol.

Persistence model

  • RuntimeStore, Store, TurnCheckpoint, TranscriptEntry, TranscriptKeyspace
  • Separates canonical transcript rows from runtime-only snapshot state.

Tools

  • ToolProvider, ToolDefinition, ToolParam, ToolResult
  • Unifies built-ins, plugin tools, and dynamic tool surfaces under one execution path.

Plugins

  • SessionPlugin, PluginFactory, PluginRegistrar, ExternalOpDef, ToolResultProjectionHook
  • Hooks span prompt shaping, stream transforms, checkpoints, turn lifecycle, external ops, and tool result projection.

CLI projection layer

  • DisplayBlock, UiResumeState, blocks_from_transcript(), apply_ui_host_effects()
  • Derives the visible transcript from semantic state while keeping live-only UI data and host actions outside transcript truth.

Prompt shaping

  • PromptContext, PromptContribution, PromptSectionOverride, PromptRenderer, PromptRequest, PromptResponse
  • Keeps prompt assembly and user prompting extensible without hardwiring every special case into the runtime loop.

REPL runtime

  • Program, Stmt, Expr, State, ToolHost, ExecutionOutcome
  • Provides a compact local language for iterative tool-driven execution.

Dependencies

The stack is intentionally practical: async runtime, HTTP transport, SQLite, terminal UI, and a thin host-extension layer instead of a larger application framework.

Dependency clusters

Most third-party code supports one of six jobs: async orchestration, provider transport, local persistence, terminal rendering, host/UI adaptation, or local execution utilities.

Runtime + transport

  • tokio drives the async runtime, process I/O, timers, and session/task coordination.
  • reqwest handles provider HTTP transport with rustls and streaming.
  • uuid, sha2, and chrono support durable ids, hashing, and timestamped session files.

Persistence + local environment

  • rusqlite is the storage backbone for session metadata, live session snapshots, and canonical transcript rows.
  • dirs, tempfile, ignore, and globset support repo and filesystem work.
  • portable-pty underpins long-lived shell execution.

Terminal presentation

  • lash-tui and crossterm power the full-screen CLI.
  • pulldown-cmark and unicode-width shape assistant markdown into terminal-safe layouts.
  • image and arboard support image/clipboard affordances around the TUI.

Host extension surface

  • lash-sansio keeps the protocol portable for non-CLI hosts.
  • lash-ui adds no renderer dependency; it depends only on the runtime contracts plus small async/JSON helpers.
  • lashlang provides a local programmable substrate without binding the whole runtime to shell scripting.

orientation

Read in this order

  1. lash/src/lib.rs, lash-ui/src/lib.rs, and lash-cli/src/main.rs for the module map and the new layer boundary.
  2. lash/src/runtime.rs and lash-sansio/src/sansio.rs for the turn contract.
  3. lash/src/plugin.rs plus lash/src/plugin_builtin/plan_mode.rs for plugin/runtime policy.
  4. lash-cli/src/interactive/mod.rs, lash-cli/src/app/mod.rs, lash-cli/src/render/mod.rs, and lash-cli/src/scratch_tui.rs only after you understand the block model and host-effect flow.

Current rule of thumb: if a piece of data affects conversation meaning, it belongs in the semantic transcript. If it is host transport, command wiring, or a visual affordance, it belongs in lash-ui or the concrete host, not in the protocol core.
