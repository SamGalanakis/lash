Host Interface
Frontend Modes
Three host modes feed into a single runtime API. Each produces ordered TurnInput items that the runtime normalizes before dispatch.

Interactive CLI

Slash commands, image paste markers, file/path references.

Headless

Single autonomous run with printed final output.

Embedded Host

Programmatic RuntimeEngine::run_turn with structured items.

Normalization
Runtime Item Normalization
Runtime validates references, canonicalizes paths, binds image blobs by id, and preserves ordered interleaving. This is the host/agent contract boundary.

Contract

TurnInput { items[], image_blobs{} } → normalized text/image stream.

Validation

Missing image ids and invalid paths fail early with runtime error events.

Agent Core
Prompt Assembly + Iterative Loop
Agent resolves provider/model, refreshes tokens, reads dynamic projection state from the active tool registry, injects only tools with inject_into_prompt=true, and appends discovery guidance for omitted tools (list_tools/search_tools, tools.<tool>).

Prompt Tooling

Visible tools split into prompt-injected subset + discoverable runtime subset from the active generation.

Context Control

Rolling window + _history / _mem for long-running sessions and sub-agents.

Capability Guidance

Prompt helper bindings + capability prompt sections come from dynamic capability definitions.

Execution Bridge
Session + Embedded Python REPL
Session spins up embedded Python, registers tool definitions/capabilities, and executes code blocks in a persistent namespace. Runtime reconfigure updates the same REPL between turns with atomic rollback semantics.

Bidirectional Signals

Message streaming, final responses, tool-call records, image attachments, and prompt requests.

State Durability

Runtime keeps agent envelope + optional REPL snapshot blob for restore/resume; reconfigure never applies mid-turn.

Tools + Delegation
Composite Tool Graph
Static providers are wrapped by DynamicToolProvider (in-process adapter), then projected by dynamic capability profiles. CLI can stage edits via /tools and /caps, with apply on next turn.

Core Tools

read_file, edit_file, shell, glob, grep, tasks, plan.

Reconfigure

All-or-nothing apply with generation checks; failures roll back and block turn dispatch.

Delegation

agent_call forks parent dynamic state per sub-agent; low tier is filtered (no write/delegation tools).

Persistence
Store, Logs, and Session State
SQLite-backed store tracks tasks/context artifacts for the session, while CLI keeps session logs and runtime state envelope that can be restored on resume.

Store

Session-specific DB under ~/.lash/sessions/*.db.

Envelope

AgentStateEnvelope: messages, iteration, token usage, task/subagent state, snapshot.

02 Dataflow Primary execution path from host input through agent loop to streamed output.

Primary Dataflow

01 Host Input

Frontend emits ordered TurnInput.items + optional image blobs.

02 Normalize + Apply

Runtime validates refs and applies pending dynamic reconfigure before turn start.

03 Agent Step

Prompt assembled from active tool/capability projection + injected tool docs.

04 REPL Exec

Python executes code blocks; bridge dispatches tool calls through dynamic adapter routing.

05 Output

Events stream back; final answer + updated envelope returned to host.

Prompt-visible inject_into_prompt=true Discoverable list_tools / search_tools / tools.<tool> Reconfigure /tools / /caps / /reconfigure Unified Items text + refs + image + skill
Architecture constraint: sub-agents share the same filesystem/worktree. Each sub-agent has an isolated fork of parent dynamic tool/capability state, but parallel branches should still avoid editing the same files concurrently.
