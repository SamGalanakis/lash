Current workspace architecture

lash/system

lash is a Rust workspace for an embeddable terminal coding agent. The current shape is a layered system: lash-sansio provides the deterministic turn protocol, lash hosts runtime, plugins, persistence, providers, and sessions, and lash-cli assembles the default application experience. Runtime construction, turn execution, provider registration, session bootstrap, stable resume identifiers, and host documentation now live behind focused module boundaries instead of the older monolithic paths.

Overview

The workspace is intentionally split between pure protocol logic, host runtime orchestration, tool/plugin composition, provider crates, and opinionated application hosting. The latest architectural shift keeps CLI-only concerns in the CLI host: resume/fork now addresses sessions by stable id or name while legacy database filenames remain an internal storage detail.

Current architectural truths

  • Runtime is no longer a single slab: construction/resume, session APIs, turn orchestration, effect execution, streaming, and child-session management are separate modules.
  • Providers have one display source: provider labels live in ProviderFactory implementations and hosts resolve them through the registry.
  • Sessions are first-class: root sessions, subagent children, forked sessions, and benchmark sessions reuse the same runtime/session manager machinery.
  • Resume is id-first: startup, /resume, and fork fallback resolve session id/name to the stored database path instead of exposing filenames as the primary user interface.
  • Dual execution remains central: standard and RLM mode share policy, plugin, provider, and persistence infrastructure.

Modules

The codebase reads best as a stack of crates with clear responsibility boundaries. Changes to prompts, tools, and execution typically pass through only one or two layers.

Protocol kernel

lash-sansio

Pure, deterministic turn protocol logic. It owns prompt template rendering, message/session model types, tool definition formatting, effects, and responses without assuming async I/O or storage.

  • session_model/prompt.rs now defines PromptTemplate, PromptSlot, and built-in prompt fragments
  • TurnMachine models request/response iteration and tool protocol behavior
  • Strongest boundary for pure protocol changes
Runtime core

lash

The architectural center. It owns sessions, stores, providers, runtime builders, session manager behavior, plugin execution, and the host-side prompt/tool surface.

  • runtime/lifecycle.rs, session_api.rs, and turn_loop.rs hold runtime construction, host APIs, and turn orchestration
  • runtime/turn_driver/ separates effect handling, tool dispatch, and provider streaming
  • runtime/session_manager/ separates the trait API, user prompts, and child-session usage forwarding
  • plugin/mod.rs and plugin/runtime_impl.rs define the extension system
  • session.rs binds plugins, tools, lashlang runtime, and per-session context surface
CLI host

lash-cli

The main application. It handles startup, provider selection, persisted session bootstrap, TUI, autonomous runs, rendering, and command-level host policy.

  • session_bootstrap.rs centralizes fresh, resume, and fork session file/store/logger setup
  • session_log.rs, resume.rs, and fork.rs keep session id/name resolution above the storage filename layer
  • host_docs.rs writes managed Markdown into the configured Lash home and injects that path through a CLI-only prompt contribution
  • bootstrap.rs assembles default plugins, providers, policies, and the runtime host
  • main.rs now picks default_prompt_template() or an autonomous-specific template
  • Richest concentration of user-facing behavior and UX details
Execution language

lashlang

The RLM programming language runtime used when the model executes persistent code across turns.

  • Separate AST/runtime evolution from host runtime concerns
  • Supports persistent variables, tool calls, print/await/cancel, and typed submit paths
  • Integrated by lash session runtime rather than by the CLI directly
Provider layer

provider crates + lash-providers-builtin

Each provider crate owns one factory and one credential/config implementation. The umbrella crate registers the first-party set for hosts that want the stock provider catalog.

  • ProviderFactory::cli_label() is the single display-label source
  • ProviderHandle stores the runtime provider object and resolves model/variant policy
  • lash-providers-builtin::register_all() removes duplicated registration blocks from hosts
First-party tool composition

lash-default-tools

Builds the standard tool/plugin bundles for different surfaces, context approaches, and execution modes.

  • Wraps filesystem, shell, fetch/search, ask/update-plan, and instruction-aware tools
  • Chooses bundles via DefaultToolSurfaceProfile
  • Feeds tools and prompt contributions into the plugin host
Subagent layer

lash-subagents

Provides the spawn_agent tool family and the policy for host-backed child sessions with capability-scoped tool visibility and separate usage accounting.

  • Creates child SessionCreateRequest values with tier-specific tool surfaces
  • Adds subagent guidance via prompt contributions
  • Bridges standard and RLM subagent semantics
Presentation support

lash-ui + lash-tui

Shared UI/state and terminal drawing support for the CLI. They reduce direct terminal complexity inside lash-cli.

  • lash-ui holds UI commands/extensions
  • lash-tui abstracts terminal rendering infrastructure
  • Both stay below the main CLI host layer
Benchmark/support

bench runners + lash-export

Support crates for evaluation and workspace automation.

  • LongCoT and LongMemEval runners target OpenAI-compatible/RLM benchmark paths
  • SWE-bench uses the default runtime/plugin stack plus built-in provider registration
  • lash-export turns stored sessions into exportable artifacts

Data Flow

At runtime the host builds a session, resolves a provider through the registry, plugins produce tools and prompt contributions, the runtime builds a prompt context, the host template renders the system prompt, and the sans-IO turn machine drives provider/tool interaction.

Prompt path after the template cutover

  • ModeExecutionPlugin::build_execution_preamble creates execution instructions, visible tools, and mode-specific prompt contributions.
  • lash-sansio::build_prompt merges prompt contributions into PromptContext.
  • PromptTemplate::render owns the final system prompt layout.
  • Plugin-provided contributions include runtime context, project instructions, plan guidance, shell guidance, and delegation notes.
  • CLI host docs are written under the active Lash home and surfaced as a prompt path without changing core runtime behavior.

Persistence path

  • Turns emit usage/events and may append plugin/session graph nodes.
  • Persistent runtimes snapshot graph state, plugin state, and checkpoints through store abstractions.
  • Resume accepts stable session ids or names, then maps them to the current SQLite filename through session metadata.
  • Child sessions reuse the same store/session manager APIs but can carry different policy, tool surfaces, and usage-source labels.

The host runtime is asynchronous and session-oriented. The same machinery handles the root CLI session, autonomous runs, subagent child sessions, and internal compaction/benchmark helper sessions.

Runtime split

Lifecycle, session API, turn loop

LashRuntime construction/resume lives in lifecycle.rs, host-facing APIs live in session_api.rs, and turn assembly/finalization lives in turn_loop.rs.

Turn driver split

Effects, tools, streaming

The turn driver coordinates the sans-IO machine while child modules own checkpoint/code effects, tool dispatch, stream transformation, debug logging, and trailing usage capture.

Standard mode

Provider-native tool loop

Uses provider transports and model tool schemas directly. The turn machine mediates repeated tool-call / response iterations until completion.

RLM mode

Persistent lashlang loop

Mode plugins initialize a lashlang runtime, build an execution preamble, and allow the model to emit executable lashlang blocks across iterations.

Background work

Executor-backed tasks

Observation/memory work and long-lived operations can run in the background through BackgroundExecutor and background runtime hosts.

Child sessions

Delegation + utility sessions

The session manager can create child sessions with fresh or inherited plugins, filtered tool surfaces, alternate policies, and distinct usage-source labels.

Prompt model

PromptTemplate + PromptSlot

The host chooses layout. Plugins provide content. This replaces the older public renderer/override story and makes prompt structure enumerable and host-owned.

  • Built-ins: MainAgentIntro, ExecutionInstructions, CoreGuidance
  • Slots: Intro, Execution, Guidance, ProjectInstructions, RuntimeContext, Environment
Runtime hosting

RuntimeCoreConfig

Central host config object for base directory resolution, prompt template, path resolution, provider factory, sanitizer, termination policy, and optional logging.

Embedding entry

EmbeddedRuntimeBuilder

Fluent builder for configuring runtimes with session IDs, policies, plugin hosts, stores, background executors, and the prompt template.

Plugin architecture

PluginFactory / SessionPlugin

Factories build session-scoped plugins. Plugins register tools, prompt contributions, history rewriters, runtime hooks, and external operations.

Provider registry

ProviderFactory + ProviderHandle

Provider factories own config deserialization, setup labels, and CLI display labels. Runtime policy carries a ProviderHandle with model, variant, readiness, and request behavior.

CLI persistence bootstrap

SessionBootstrap

Fresh, resume, and fork startup paths share store, logger, session name, persisted config, initial graph setup, and id/name-to-file resolution through narrow CLI-host helpers.

Instruction sourcing

InstructionSource

Abstracts system and context-aware project instructions. The default implementation reads AGENT/AGENTS/CLAUDE files, but embedders can swap in their own source.

Persistence surface

RuntimeStore + session graph

Store abstractions persist session state, graph nodes, blobs, checkpoints, live resume snapshots, and plugin artifacts with optional SQLite backing.

Dependencies

The workspace is mostly self-contained, but it relies on a predictable set of libraries and services for provider access, terminal UI, persistence, HTTP, and async orchestration.

Core libraries

Tokio, Serde, Tracing

Tokio powers async runtime and background work, Serde drives all transport/config/state serialization, and tracing provides runtime/debug logging.

Terminal stack

Clap, Crossterm, portable-pty

The CLI and shell tools rely on Clap for argument parsing, Crossterm for terminal control, and portable-pty for interactive subprocess execution.

Persistence

SQLite-backed optional store

When the store feature is enabled, runtime/session persistence and blob retention can use SQLite through the store layer.

External integrations

Providers, Tavily, MCP

Provider adapters drive LLM calls. The built-in provider crate registers Anthropic, Codex, Google OAuth, and OpenAI-compatible factories for hosts that want the stock catalog. Optional web search and MCP attachment add broader external tool integration surfaces.

CLI-owned knowledge

Host docs under Lash home

The CLI writes small managed Markdown docs into the configured Lash home and tells the model where to read them. That keeps self-knowledge host-specific instead of adding a core runtime tool.
