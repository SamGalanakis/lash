Architecture / 2026-04-14

lash/system

lash is a Rust workspace for running a graph-backed, plugin-driven coding assistant across three layers: a pure turn machine in lash-sansio, an embeddable runtime in lash, and an opinionated host in lash-cli. The current architecture is graph-first and append-only: committed history lives in SessionGraph, durable runtime state is checkpointed behind blob refs, live resume is stored as base checkpoint + graph delta, and default tool bundles are opted into explicitly through lash-default-tools.

Verification
Current checks are green cargo check --workspace and cargo test -q -p lash -p lash-cli -p lash-default-tools both passed while this page was updated.
Current posture
Host-owned policy, core-owned semantics The runtime keeps replay, session graph, checkpoint ordering, execution modes, and plugin semantics. Hosts choose transport, background execution, persistence backend, UI, and tool bundles.
Latest rewrite
Graph-first persistence cutover The CLI no longer owns direct checkpoint/head mutation logic. It now persists through RuntimePersistenceState and Store methods, with only UI-resume policy left at the host layer.

Overview

The workspace is eight Rust packages wide, with the assistant runtime concentrated in lash-sansio, lash, lash-default-tools, and lash-cli. The system now favors immutable graph history, checkpoint refs, and host-provided execution policy over ad hoc snapshot mutation.

Modules

The workspace is deliberately layered. The crates below are the modules that matter most when changing request flow, persistence, tool wiring, or UI behavior.

lash-sansio

Pure turn machine and prompt model. No async runtime, no filesystem policy, no persistence backend, and no host UI logic.

  • sansio/mod.rs: effect-driven turn state machine
  • session_model/: messages, prompts, execution metadata
  • Owns the deterministic protocol layer that lash orchestrates

lash

Embeddable runtime. Owns session graph semantics, persistence contracts, host services, plugin orchestration, tool dispatch, and execution modes.

  • runtime/: host config, turn driver, session manager
  • session_graph.rs: append-only conversation and plugin graph
  • store.rs: checkpoint blobs, session head, live resume, usage deltas

lash-default-tools

Optional first-party tool bundle crate. Keeps default shell, file, search, web, state, and context-approach plugins out of the core constructor path.

  • DefaultToolSurfaceProfile maps execution mode, context approach, interactivity, and web enablement to bundle selection
  • Owns Rolling History / OM bundle selection on the default host path

lash-cli

Opinionated host. Boots providers, builds the plugin host, runs interactive or autonomous sessions, saves UI state, manages resume/fork, and exposes benchmarking/profiling utilities.

  • bootstrap.rs: host assembly
  • turn_runner.rs + persistence.rs: shared turn lifecycle and host-side UI persistence policy
  • interactive/, autonomous.rs, resume.rs, fork.rs

lash-ui + lash-tui

Rendering support crates. lash-tui is the low-level terminal drawing layer, while lash-ui provides UI effect abstractions reused by the CLI.

  • Separated from the runtime so core remains host-agnostic
  • Lets the CLI evolve without leaking UI assumptions into lash

lashlang

The RLM language runtime used by the assistant’s executable mode. It is kept separate from the assistant runtime so the graph/plugin/store architecture does not depend on language internals.

  • Lexer, parser, AST, and runtime live here
  • Embedded via lash::embedded and lash::session

Data Flow

Request flow now has two clean boundaries: hosts build policy and services, then lash runs the graph-backed turn. Persistence and live resume use checkpoint refs and graph deltas instead of full transcript snapshots.

Key current flows

  • Committed turn: runtime exports RuntimePersistenceState, store stamps snapshots and writes head/checkpoint state, CLI only saves UI resume state around that core call.
  • Interrupted turn: live resume writes a base graph plus LiveResumeDelta instead of a full transcript snapshot.
  • Fork without runtime: CLI materializes the latest coherent live graph, forks its current path, and carries over the right checkpoint or live delta turn state.
  • OM background work: plugin jobs run through the background executor seam, then append memory nodes back onto the session graph.

Execution

The runtime and CLI now share one turn runner. Interactive and autonomous differ in sinks and UI policy, not in core turn execution semantics.

Turn lifecycle

Interactive path

The TUI uses interactive/mod.rs, interactive/runtime.rs, and the app/view/render stack. It still owns UI resume state, prompt overlays, queue editing, and rendering, but it no longer owns checkpoint mutation rules.

Autonomous path

autonomous.rs runs the same turn runner with a stdout-oriented sink. Interactive-only tools are filtered out in bootstrap before the host is assembled.

Branching and restore

resume.rs, resume_snapshot.rs, and fork.rs load the live snapshot first when it is safe, then fall back to the committed head. The runtime itself stays authoritative once loaded into memory.

Performance tooling

runtime_perf.rs and scripts/profile_runtime.py provide no-inference runtime profiling, including long-chat synthetic turns and optional dhat heap captures.

Dependencies

The workspace keeps the pure kernel small, then layers async runtime, persistence, UI, and network dependencies on top where they belong.

External dependency map
flowchart LR Sansio["lash-sansio"] --> Serde["serde / serde_json / uuid / base64"] Core["lash"] --> Tokio["tokio / tokio-util"] Core --> Sqlite["rusqlite (feature gated)"] Core --> Net["reqwest / oauth / provider adapters"] Core --> Search["fff-search / globset / ignore / regex"] Core --> Pseudo["portable-pty / tempfile / sha2 / tracing"] CLI["lash-cli"] --> TUI["crossterm / lash-tui / lash-ui"] CLI --> CLIExt["clap / arboard / tracing-subscriber / stats_alloc / dhat"] DefaultTools["lash-default-tools"] --> Core

Key runtime libraries

  • tokio and tokio-util drive async execution and cancellation.
  • rusqlite backs the built-in store behind the sqlite-store feature.
  • reqwest underpins provider and web tool HTTP integration.
  • fff-search powers the indexed grep backend; the workspace can also enable the fff-zlob feature.
  • portable-pty supports shell tool execution.

Intentional boundaries

  • lash-sansio has none of the async UI / store / tool runtime dependencies.
  • First-party tools are not built into lash; they are composed through lash-default-tools.
  • The CLI is where terminal rendering, clipboard behavior, profiling flags, and setup/auth flows belong.
