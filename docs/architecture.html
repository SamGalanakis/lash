Current workspace architecture

lash/system

lash is a terminal AI coding agent written from scratch in Rust. The codebase is organized as a layered workspace: a deterministic sans-IO turn engine, an async host runtime, first-party tool bundles, delegation tooling, dedicated terminal/UI crates, and a CLI host that drives both interactive and autonomous operation.

Overview

The workspace is broad but intentionally modular. Core protocol logic stays pure in lash-sansio, orchestration and persistence live in lash, and the end-user experience is assembled in lash-cli with separate UI, TUI, tool-bundle, delegation, benchmarking, and build-support crates.

What matters right now

  • Host/runtime split is real: lash-cli is opinionated, but the runtime is still embeddable through lash.
  • Plugins are first-class: tools, prompt contributions, context approaches, planning, and delegation all hook in via plugin factories and session plugins.
  • Execution is dual-mode: the same runtime supports provider-native tool calling and stateful lashlang execution.
  • Persistence is graph-based: committed sessions, checkpoints, live resume state, and plugin snapshots are stored through the runtime/store layer.

Modules

The workspace is best understood as a set of architectural layers rather than a pile of files. Each crate owns a clean responsibility boundary, and most changes should stay within one of these layers.

Host layer

lash-cli

The end-user application and the most opinionated host. It owns startup, provider/config setup, interactive terminal experience, autonomous runs, resume/fork behavior, and benchmark entrypoints.

  • Consumes the core runtime rather than redefining it
  • Assembles the default plugin and tool surface during bootstrap
  • Contains the richest host-specific policy in the workspace
Runtime layer

lash

The architectural center of gravity. It turns policy, plugins, tools, provider access, and persistence into a reusable agent runtime that both the CLI and embedders can use.

  • Coordinates turns, sessions, persistence, background work, and providers
  • Exposes the main public APIs for embedding lash elsewhere
  • Keeps host concerns above the pure protocol layer
Protocol kernel

lash-sansio

The pure state-machine layer. It models messages, prompts, effects, responses, and execution modes without assuming a filesystem, terminal, database, or network stack.

  • Defines deterministic turn behavior
  • Forms the stable core underneath all hosts
  • Provides the cleanest place for protocol-only changes
Execution language

lashlang

The RLM DSL runtime used when the assistant runs executable local programs across turns. It is intentionally separate from the host runtime so the language can evolve without tangling host orchestration code.

  • Owns parsing, runtime state, values, and tool-host integration
  • Powers the persistent programmable mode of the agent
  • Acts as an optional backend rather than the only execution model
Plugin and tool composition

lash-default-tools + lash-delegate-tools

These crates keep tool-surface composition out of the core runtime. One packages the default first-party bundles; the other adds controlled child-session delegation.

  • Default bundles map runtime policy to concrete tool/plugin sets
  • Delegation is modeled as a plugin/tool surface, not special-case app logic
  • This keeps the runtime modular and embedders flexible
Presentation layer

lash-ui + lash-tui

The workspace separates UI behavior from low-level terminal drawing. That keeps rendering and host interaction concerns from leaking into the runtime.

  • lash-ui models commands, shortcuts, and host-side UI effects
  • lash-tui handles terminal drawing primitives and screen mechanics
  • Together they support a richer CLI without coupling the core runtime to terminal details
Support layer

Bench runner + scripts

Operational tooling, profiling, release helpers, and benchmark harnesses live outside the production runtime path. This keeps evaluation and maintenance work visible without inflating the core architecture.

  • Benchmarking and profiling stay explicit and separable
  • Workspace automation does not distort runtime design
  • Useful context when performance or release processes are under change

Current system flows

  • Interactive turns: the TUI edits input and shows streamed events, but the runtime still owns the protocol, tool execution, and persisted session state.
  • Autonomous turns: lash -p uses the same runtime path with stdout-oriented sinks and a filtered tool surface.
  • Delegation: delegated work is just another tool surface, bound to session context and constrained by policy before child sessions run.
  • Observational memory: background plugin work observes history and feeds memory artifacts back into the graph instead of changing the core turn protocol.

Execution

The runtime is effect-driven. Hosts supply services and presentation policy; lash coordinates turn execution; lash-sansio decides what must happen next. Standard mode and RLM mode differ in how tool/code execution is expressed, not in how sessions are hosted.

CLI startup

Bootstrap path

lash-cli/src/bootstrap.rs wires providers, session policy, default tool bundles, delegate tools, UI extras, and runtime configuration before any turn runs.

Mode handling

Interactive vs autonomous

Interactive mode keeps terminal and UI state alive; autonomous mode runs one or more turns and prints or records results. Both rely on the same core runtime APIs.

Tool execution

Static + dynamic surfaces

Tools are exposed via ToolProvider. Some are static first-party tools; others come from plugin sessions, MCP attachments, or delegated child-session support.

Benchmark paths

Performance instrumentation

runtime_perf, UI benchmark flags, and scripts under scripts/ keep runtime and UI profiling available without contaminating the main execution model.

Abstractions

The most important abstractions in lash are architectural, not just type-level. The system is designed around a few durable seams that let protocol logic, host orchestration, tool composition, UI behavior, and persistence evolve without collapsing into one giant application layer.

Separation of concerns

Protocol vs host

The deepest seam in the system is the split between the pure turn protocol and the host runtime that satisfies it. That is what allows lash-sansio to stay deterministic while lash handles I/O, async work, provider calls, and storage.

  • Protocol logic can stay testable and portable
  • Host features can grow without contaminating the kernel
  • Embedders can reuse the runtime or go lower-level as needed
Composition model

Plugins over hardcoding

Planning, memory, tool registration, prompt shaping, delegation, and session-scoped behavior are treated as composable plugins. That keeps the runtime from becoming a single giant switchboard for every feature.

  • Features can attach at session scope instead of being globally baked in
  • Hosts can assemble different surfaces for interactive and autonomous runs
  • New behavior usually starts as a plugin seam before it becomes a core concern
Execution model

One runtime, two execution styles

The architecture supports both provider-native tool calling and persistent lashlang execution. The important abstraction is that both are hosted by the same runtime and session model, rather than becoming two separate products.

  • standard mode is closer to provider-native tool orchestration
  • rlm mode adds a programmable local execution layer
  • The surrounding session, persistence, and plugin model remains shared
State model

Graph-backed sessions

Conversation state is not treated as a single flat transcript. The runtime architecture is built around graph-like session state, checkpoints, snapshots, and plugin-owned artifacts.

  • Better fit for resume, branching, retries, and background work
  • Lets plugin-produced state live alongside user/assistant turns
  • Supports richer lifecycle handling than transcript-only designs
Tool surface

Capability surfaces, not ad hoc commands

Tools are exposed as a coherent capability layer. The default host builds a standard surface, but the architecture allows static tools, dynamic tools, MCP-style integrations, and delegated child-session workers to share the same general model.

  • Default bundles provide consistency
  • Dynamic and delegated tools extend capability without changing the kernel
  • Tool exposure is part of host composition, not an afterthought
Presentation boundary

UI behavior separated from rendering

The workspace keeps terminal rendering and UI semantics out of the runtime. That makes the CLI richer without making the core agent architecture terminal-specific.

  • UI commands/effects live separately from screen drawing primitives
  • The runtime can stream events without owning how they are displayed
  • Presentation remains a host concern instead of a protocol concern

How to reason about changes

  • If the change is about turn semantics, think protocol-first.
  • If it is about composition or feature policy, think plugin-first.
  • If it is about state durability or resume, think graph/store-first.
  • If it is about user interaction, think host/UI-first.

Why this matters

These seams are what make the repo manageable as it grows. They are more important than memorizing individual type names, because they tell you where a new capability belongs and what it should avoid coupling to.

Dependencies

The dependency story is less about specific crate names and more about where outside complexity enters the system. lash is structured so that expensive, stateful, and platform-specific concerns stay in outer layers, while the kernel remains small and portable.

Kernel dependency posture

Keep the center light

The protocol core depends on foundational serialization and model types, but avoids taking on async runtimes, terminal concerns, filesystem policy, or storage engines.

  • This preserves portability and testability
  • It also makes the core easier to reason about independently of the app host
Runtime dependency posture

Own the real-world integration layer

The runtime is where external systems enter: network providers, persistence, search and file access, PTYs, clocks, caches, and background execution.

  • This is the right place for I/O-heavy dependencies
  • It lets the kernel stay conceptually clean
Host dependency posture

Terminal and UX stay at the edge

The CLI owns terminal, clipboard, rendering, markdown formatting, update checks, and user-facing instrumentation. Those dependencies are intentionally not pulled into the core protocol layer.

  • The host can evolve quickly without destabilizing the kernel
  • Alternative hosts remain possible because UI assumptions are not baked into the runtime core
Optionality

Features gate heavyweight integrations

Some expensive dependencies are optional or conditional, especially around persistence and tool implementations. That keeps the embeddable core smaller and lets hosts opt into richer behavior only when needed.

  • SQLite-backed persistence is feature-gated
  • Tooling-heavy integrations stay out of the minimum core
Operational ecosystem

Benchmarks and profiling are adjacent

Benchmark harnesses, profiling scripts, and reporting tools are part of the repo’s engineering system, but they are intentionally adjacent to the main runtime rather than tangled into it.

  • Performance work stays visible
  • Production execution paths stay comparatively clean
Architectural principle

Push complexity outward

The repo consistently pushes network, platform, terminal, and tooling complexity away from the center. That principle is one of the clearest organizing ideas in the codebase.

  • Pure center, integrated edge
  • Reusable runtime, opinionated host
  • Composable features, constrained coupling

Dependency heuristics

  • If a new dependency is platform-specific, it probably belongs in the host or outer runtime layers.
  • If it is needed only for one optional feature, prefer gating or isolating it.
  • If it affects deterministic turn semantics, resist putting it in the kernel.
  • If it is purely for docs, profiling, or benchmarks, keep it adjacent rather than central.

Takeaway

The dependency architecture is designed to protect the center of the system. That is why the repo can support a rich CLI, a flexible runtime, delegated workers, and a custom DSL without turning the entire workspace into one tightly coupled crate.

Where to change behavior

By concern, not by file hunt

Most work naturally falls into one architectural bucket. Start with the owning crate and only then drill into specific files.

  • Protocol behavior: start in lash-sansio
  • Runtime orchestration and persistence: start in lash
  • Default tool surface: start in lash-default-tools
  • Delegated workers: start in lash-delegate-tools
  • Interactive UX: start in lash-cli, then lash-ui/lash-tui as needed
