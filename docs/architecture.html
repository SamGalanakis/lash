lash architecture dossier

Transcript/Core

lash is a Rust terminal coding agent built as a layered system: a sans-IO turn machine in lash-sansio, an async host runtime in lash, a full-screen TUI in lash-cli, and a small local execution language in lashlang. The current storage model is intentionally centralized: SQLite persists one canonical semantic transcript, and the CLI derives visible history from that transcript instead of saving a second UI transcript.

What lash is

A terminal-first coding agent that reads and writes repositories, runs local tools, calls remote model providers, tracks long-lived sessions, and exposes enough protocol structure to support plugins, delegation, autonomous mode, and durable resume.

  • lash-sansio owns turn control, message types, and prompt shaping contracts.
  • lash owns runtime orchestration, tool execution, plugins, providers, and persistence.
  • lash-cli owns startup, interactive mode, autonomous mode, and terminal rendering.
  • lashlang owns the small DSL used by REPL execution mode.

What changed architecturally

Session persistence is now centralized around semantic transcript rows in SQLite. The store keeps a slim agents runtime snapshot for iteration counters, token usage, plugin snapshot state, and UI-only live resume hints, but it no longer stores a second transcript-shaped UI payload.

  • load_session() reads transcript rows, reconstructs Message[] and ToolCallRecord[], then derives display blocks.
  • load_picker_info() queries the transcript directly for first prompt and message count.
  • UiResumeState is an overlay for live assistant text, plugin panels, and streaming tails, not transcript truth.

CLI-resume path

  • session_log.rs loads the canonical transcript and derives DisplayBlock history.
  • ui_resume.rs reads and writes the small live-only overlay embedded in agents.config_json.
  • resume.rs and fork.rs stitch store state back into the app/runtime.

Current hotspots

  • lash/src/runtime.rs — orchestration, streaming, checkpoints, child sessions.
  • lash-cli/src/ui.rs — terminal rendering, overlays, scroll calculations.
  • lash-cli/src/app.rs — transcript projection, turn finalization, block management.

Deliberate split

  • The protocol core does not know about ratatui, rusqlite, or reqwest.
  • The store does not persist pre-rendered UI blocks as transcript truth.
  • The CLI owns presentation and live affordances, not canonical conversation semantics.

Data Flow

How session data moves now that the transcript is centralized: semantic rows go in, message/tool projections and display blocks come out.

flow

Canonical transcript pipeline

The storage contract is semantic. UI history is derived on load and only lightly overlaid with live resume state.

flowchart LR U["User input or resume"] --> SL["session_log load path"] DB[("SQLite store")] --> SL SL --> PROJ["Rebuild messages and tool calls"] PROJ --> BLOCKS["Project display blocks"] UISTATE["UiResumeState overlay"] --> BLOCKS BLOCKS --> TUI["Visible transcript in ui.rs"] TUI --> RT["LashRuntime"] RT --> TM["TurnMachine"] TM -->|llm call| LLM["Provider adapters"] TM -->|tool or exec| TOOLS["Tools, plugins, lashlang"] LLM --> TM TOOLS --> TM TM --> SNAP["AgentStateEnvelope"] SNAP --> KS["message and tool_call keyspaces"] KS --> DB

Single source of truth

  • transcript_entries is the canonical persisted transcript table.
  • Two keyspaces matter today: message and tool_call.
  • load_session() always rebuilds blocks from semantic transcript rows.
  • load_picker_info() also queries transcript rows directly rather than reading a UI cache.

What is not transcript truth

  • agents.config_json carries plugin snapshot and task/runtime metadata.
  • UiResumeState stores only live affordances that cannot be derived from completed transcript rows.
  • Examples: streaming tail lines, plugin panels, and partial assistant text during an interrupted turn.
resume path

Load projection steps

  1. Open the session database and fetch transcript rows for root.
  2. Rebuild Message[] and ToolCallRecord[] from the transcript.
  3. Project those semantic records into DisplayBlock values.
  4. Overlay UiResumeState for live assistant text or plugin panels when needed.
write path

Persist steps

  1. Collect current runtime state into AgentStateEnvelope.
  2. Serialize runtime-only config into agents.config_json.
  3. Derive canonical transcript keyspaces with semantic_transcript_keyspaces().
  4. Replace the stored keyspaces atomically for the root agent.

Execution

Operational flow inside a session: resume, turn execution, checkpoints, and the two execution modes the runtime can host.

sequence

Resume and turn lifecycle

The runtime loop is effect-driven, but session startup and final persistence sit on either side of that loop.

sequenceDiagram participant CLI as lash-cli participant STORE as Store participant RT as LashRuntime participant TM as TurnMachine participant LLM as Provider participant TOOLS as Tools / Plugins / lashlang CLI->>STORE: open session DB STORE-->>CLI: transcript_entries + agents + session_meta CLI->>CLI: blocks_from_transcript() CLI->>RT: submit prompt / resume turn loop until turn completes RT->>TM: poll / handle response alt LLM effect TM-->>RT: Effect::LlmCall RT->>LLM: request / stream LLM-->>RT: response chunks + usage RT->>TM: Response::LlmComplete else Tool or exec effect TM-->>RT: Effect::ToolCalls or Effect::ExecCode RT->>TOOLS: execute tool calls or lashlang program TOOLS-->>RT: results / code output RT->>TM: Response::ToolResults or Response::ExecResult else Checkpoint TM-->>RT: Effect::Checkpoint RT->>STORE: save_turn_checkpoint() RT->>TM: Response::Checkpoint end end RT->>STORE: save final root snapshot + transcript keyspaces RT-->>CLI: assembled turn + events

Standard mode

  • The runtime passes tool schemas to the provider.
  • The model emits native tool calls directly.
  • The host executes those calls and feeds ToolResult data back into the machine.

REPL mode

  • The prompt includes rendered tool docs instead of provider tool schemas.
  • The model emits fenced lashlang programs.
  • The host executes those programs locally and streams code/output records back into the transcript.

Fork and delegation

  • Child sessions copy semantic transcript rows from the parent store.
  • Runtime state is restored separately from transcript truth.
  • The same persistence primitives power resume, fork, and managed child-session work.

Checkpoint behavior

The runtime persists during turn execution via save_turn_checkpoint(), writing a durable agent snapshot and canonical transcript keyspaces. That lets interrupted turns resume without inventing a separate transcript format.

Finalization behavior

After the plugin finalize step, the runtime saves the latest semantic state again. The CLI then reconciles any live assistant overlay against the finished assistant message text so completed output always follows canonical messages.

Abstractions

The contracts that keep the system modular: the machine/runtime boundary, the store boundary, and the plugin/tool surfaces.

Turn protocol

  • TurnMachine, Effect, Response, TurnMachineConfig
  • Defines the host/core boundary. No direct I/O happens inside the machine.

Runtime host

  • LashRuntime, RuntimeHostConfig, EventSink, AssembledTurn
  • Owns provider transport, tool dispatch, session lifecycle, and checkpoint coordination.

Persistence model

  • RuntimeStore, Store, TurnCheckpoint, TranscriptEntry, TranscriptKeyspace
  • Separates canonical transcript rows from runtime-only snapshot state.

Tools

  • ToolProvider, ToolDefinition, ToolParam, ToolResult
  • Unifies built-ins, plugin tools, and dynamic tool surfaces under one execution path.

Plugins

  • SessionPlugin, PluginFactory, PluginRegistrar, ToolResultProjectionHook
  • Hooks span prompt shaping, stream transforms, checkpoints, turn lifecycle, and tool result projection.

CLI projection layer

  • DisplayBlock, UiResumeState, blocks_from_transcript(), apply_ui_resume_state_to_blocks()
  • Derives the visible transcript from semantic state while keeping live-only UI data outside transcript truth.

Prompt shaping

  • PromptContext, PromptContribution, PromptSectionOverride, PromptRenderer
  • Keeps prompt assembly extensible without hardwiring every special case into the runtime loop.

REPL runtime

  • Program, Stmt, Expr, State, ToolHost, ExecutionOutcome
  • Provides a compact local language for iterative tool-driven execution.

Dependencies

The stack is intentionally practical: async runtime, HTTP transport, SQLite, terminal UI, and a few focused utilities rather than a framework tower.

Dependency clusters

Most third-party code supports one of five jobs: async orchestration, provider transport, local persistence, terminal rendering, or local execution utilities.

flowchart TD W[lash workspace] A[Async + concurrency tokio tokio-util async-trait futures-util] N[Provider + network reqwest oauth adapters semver] S[Store + filesystem rusqlite dirs chrono ignore globset tempfile] T[Terminal UI ratatui crossterm pulldown-cmark unicode-width image arboard] X[Execution helpers portable-pty regex similar pdf-extract sha2 uuid] W --> A W --> N W --> S W --> T W --> X

Runtime + transport

  • tokio drives the async runtime, process I/O, timers, and session/task coordination.
  • reqwest handles provider HTTP transport with rustls and streaming.
  • uuid, sha2, and chrono support durable ids, hashing, and timestamped session files.

Persistence + local environment

  • rusqlite is the storage backbone for session metadata, agent snapshots, and canonical transcript rows.
  • dirs, tempfile, ignore, and globset support repo and filesystem work.
  • portable-pty underpins long-lived shell execution.

Terminal presentation

  • ratatui and crossterm power the full-screen CLI.
  • pulldown-cmark and unicode-width shape assistant markdown into terminal-safe layouts.
  • image and arboard support image/clipboard affordances around the TUI.

Embedding surface

  • lash-sansio keeps the protocol portable for non-CLI hosts.
  • lashlang provides a local programmable substrate without binding the whole runtime to shell scripting.
  • The plugin system keeps many first-party behaviors outside the core runtime loop.

orientation

Read in this order

  1. lash/src/lib.rs and lash-cli/src/main.rs for module map and entrypoints.
  2. lash/src/runtime.rs and lash-sansio/src/sansio.rs for the turn contract.
  3. lash/src/store.rs, lash-cli/src/session_log.rs, and lash-cli/src/app.rs for transcript + UI projection behavior.
  4. lash-cli/src/ui.rs only after you understand the block model and state flow.

Current architectural rule of thumb: if a piece of data affects conversation meaning, it belongs in the semantic transcript. If it only helps the CLI resume a live-but-not-final visual affordance, it belongs in UiResumeState.
