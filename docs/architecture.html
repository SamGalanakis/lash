lash architecture dossier

System/Map

lash is a Rust terminal coding agent split into a pure turn machine, an async runtime host, a full-screen CLI/TUI, and a small DSL runtime for persistent local execution. The architecture is intentionally layered: provider I/O and tool execution live in the host, conversation control lives in the sans-IO machine, and presentation stays in the CLI crate.

What the system is

A terminal-first AI coding agent that can read and write files, run shell commands, search locally and on the web, plan work, delegate to child sessions, and resume durable session state. The repo is structured so hosts can embed the protocol logic without inheriting CLI or I/O concerns.

  • lash-sansio is the deterministic turn machine.
  • lash is the async host/runtime, plugin system, provider integration, and tool execution layer.
  • lash-cli is the interactive application and autonomous single-shot entrypoint.
  • lashlang is the local DSL/runtime used by REPL execution mode.
  • xtask is the minimal workspace helper.

Architectural bias

The project strongly separates policy from effects. The machine decides what happens next by emitting effects; the runtime decides how to fulfill them. That split keeps the core testable while still allowing a large host feature surface: SQL persistence, shell execution, HTTP providers, TUI rendering, and delegated child sessions.

Data Flow

How a user prompt becomes tool execution, provider calls, state updates, and a rendered answer.

flow

Prompt-to-output path

The host runtime wraps the turn machine and repeatedly fulfills emitted effects until a turn completes.

flowchart LR U[User / CLI event] --> B[lash-cli bootstrap + app + interactive] B --> R[lash runtime session policy + prompt assembly] R --> TM[TurnMachine in lash-sansio] TM -->|LLM effect| P[Provider adapter reqwest / oauth / streaming] P --> TM TM -->|Tool effect| T[Tool dispatch built-ins + plugins] T --> TM TM -->|Checkpoint / sleep / state| S[Session store SQLite + snapshot types] S --> TM TM --> A[Assembled turn + summaries] A --> UI[lash-cli UI / stdout]

Mode split

The branch point is execution mode. standard sends tool schemas to the provider and consumes native tool calls. repl omits provider tool schemas, exposes tool docs in prompt text, and executes returned lashlang code locally.

Persistence loop

Runtime state is wrapped in AgentStateEnvelope and persisted via store.rs. That powers session resume, history export, retries, child session delegation, and durable checkpoints between turns.

Single turn lifecycle

The runtime loops on machine effects until the turn is complete or an error policy terminates it.

Standard mode

  • Tool schemas are converted into provider tool specs.
  • The model emits native tool calls in a single response.
  • The host executes those calls, often concurrently, then feeds results back into the machine.
  • This is the default path in the CLI.

REPL mode

  • Prompt contains rendered tool docs instead of provider tool schemas.
  • The model emits fenced lashlang programs.
  • The host runs those programs locally and streams code/output records.
  • This gives the agent persistent local control flow and scratch state across turns.

Abstractions

The important contracts that keep the runtime composable instead of collapsing into one giant host object.

Turn protocol

  • TurnMachine, Effect, Response, TurnMachineConfig
  • Defines the host/core boundary. The machine never performs I/O directly.

Tool contracts

  • ToolProvider, ToolDefinition, ToolParam, ToolResult
  • Allows built-in tools, plugin tools, and dynamic tool surfaces to share one execution path.

Session + runtime

  • SessionManager, SessionHandle, SessionCreateRequest, SessionSnapshot
  • Separates long-lived session ownership from per-turn execution.

Prompt shaping

  • PromptContext, PromptContribution, PromptSectionOverride, PromptRenderer
  • Lets plugins and the CLI alter prompts without rewriting the runtime loop.

Plugins

  • PluginFactory, SessionPlugin, RuntimeServices, hook contexts in plugin.rs
  • Planning, history, prompt context, tool result projection, and external ops all hang off this layer.

DSL runtime

  • Program, Stmt, Expr, Value, runtime state in lashlang
  • Provides the programmable local execution substrate for REPL mode.

Operational note

There are no long-lived external services beyond LLM and optional web APIs. Most complexity is local: protocol flow, terminal rendering, tool execution, and durable session state.
