03 Runtime Core Prompt assembly, tool projection, and runtime turn orchestration.
Runtime Core
Prompt Assembly + Sans-IO Turn Machine
LashRuntime resolves provider/model, refreshes tokens, reads dynamic projection state from the active tool registry, injects only tools with inject_into_prompt=true, and appends discovery guidance for omitted tools (search_tools). Prompt assembly is execution-mode aware: the same tool identity is projected into REPL help/docs or provider-native tool schemas depending on the active mode. Both execution paths are driven by a single synchronous state machine (TurnMachine in lash/src/sansio.rs) that emits I/O effects (LlmCall, ToolCall, ExecCode, Sleep) for the async host driver to fulfil.

Prompt Tooling

Visible tools split into prompt-injected subset + discoverable runtime subset from the active generation.

Execution Mode

repl and standard use different model-output contracts, but both flow back into the same AssembledTurn host result.

Context Control

Batched context folding with soft/hard watermarks plus search_history / search_mem tool calls for long-running sessions and sub-agents.

Capability Guidance

Prompt helper bindings + capability prompt sections come from dynamic capability definitions.

Core Architecture
Sans-IO Turn Machine
TurnMachine (lash/src/sansio.rs) is a synchronous state machine that owns all protocol logic for a single turn. The async host driver (LashRuntime in lash/src/runtime.rs) is a thin loop that calls poll_effect() to drain effects and handle_response() to feed results back. This separation makes the core turn logic independently testable without a tokio runtime.

Host Loop Contract

The host repeatedly calls poll_effect(), which returns Some(Effect) for each pending effect or None when the host must fulfil an outstanding effect before more become available. After fulfilling an effect, the host calls handle_response(response) to advance the machine. Effect::Emit events are fire-and-forget (no response needed). Effect::Done signals termination.

Effect → Response Pairs

LlmCall → LlmComplete (full response or streaming final).
CancelLlm — fire-and-forget, cancels an in-progress stream.
ToolCall → ToolResult (native tool execution).
ExecCode → ExecResult (REPL sandbox execution).
Sleep → Timeout (retry backoff).
Emit(AgentEvent) — fire-and-forget progress event.
Done — terminal, carries final messages and iteration count.

Internal States

PrepareIteration → assembles prompt, applies context folding, emits LlmCall.
WaitingLlm → awaiting provider response (REPL mode processes streaming deltas via handle_llm_delta).
WaitingTools → standard mode, parallel tool calls in flight.
WaitingExec → REPL mode, sandbox executing a code block.
WaitingRetry → backoff sleep before LLM retry.
Finished → terminal.

Owned Protocol Logic

Prompt composition, context folding (soft/hard watermarks), LLM retry with exponential backoff, fence parsing (REPL <repl> block detection mid-stream), tool call dispatch and result assembly, turn limits, malformed-output detection, and termination policy — all live in the synchronous machine. The host only performs I/O.

04 Lashlang The REPL language, execution VM, and the host contract that lets lash treat code execution as a first-class runtime path.
Execution Contract
Language Pipeline
lashlang is a standalone crate. The host can use lex, parse, compile_program, execute_compiled, and profile_compiled directly, or the convenience execute wrapper that turns parse and runtime failures into a single ExecuteError boundary.

Lexer + Parser

lexer.rs tokenizes a compact surface language with reserved words like call, observe, finish, and parallel. parser.rs builds an AST over assignments, control flow, records, lists, ternaries, tool calls, and completion.

Compiler

The compiler in runtime.rs lowers AST nodes into a compact chunk/slot form. It resolves builtins, tracks locals through slot tables, and recognizes the dedicated parallel-call execution shape before the VM runs.

Execution

The VM returns ExecutionOutcome::Continued or ExecutionOutcome::Finished(Value). Profiling is optional and produces instruction/builtin timing stats without changing the execution contract.

Parallel Rules

parallel { ... } can execute independent branches concurrently. Branches may only merge if they assign disjoint names, and finish is rejected inside a parallel branch.

Error Model

Parse errors expose precise spans and expected-token context. Runtime failures stay compact: unknown names, type/value errors, bad builtin usage, non-list iteration, or illegal parallel control flow.

Test Surface

The crate is covered by both example-style language tests and property tests for parser stability, snapshot round-trips, tool result shape, truthiness, builtins, and parallel behavior.

observe summary

Value Model
Values, Records, and Host Calls
The runtime uses a deliberately small value model: scalars, lists, and symbol-backed records. Tool calls are bridged through the host trait rather than being baked into the VM, which keeps the language reusable outside the lash CLI.

Values

Value covers null, bool, number, string, list, and record. The same type is used for locals, builtin inputs, tool outputs, and finish results.

Records

Record preserves insertion order while also indexing keys through interned symbols. That keeps field access fast and snapshots deterministic.

ToolHost

ToolHost::call executes host tools. ToolHost::observe is optional and defaults to no-op, which lets embedders ignore observation without forking the runtime.

Tool Result Shape

Host tool failures do not throw through the surface language. Tool calls resolve to records like { ok: true, value } or { ok: false, error }.

Builtins

Builtins cover formatting, string/list helpers, indexing, JSON conversion, numeric coercion, truthiness-aware conditionals, and utility helpers like len, slice, and format.

Truthiness

Control flow uses bounded truthiness rules pinned by tests, rather than inheriting Python or JavaScript semantics by accident.

Host Contract
trait ToolHost
call(name, args: Record) -> Result<Value, ToolHostError>
observe(value: &Value) -> Result<(), ToolHostError>

tool result convention

Host Integration
REPL Turn Cycle Inside Lash
Lash embeds lashlang as one execution mode. The session exposes the active tool surface, executes at most one <repl> block per assistant turn, and reinjects observations, tool results, and runtime errors as hidden feedback for the next internal step.

Contract Rules

<repl> blocks execute immediately when </repl> is reached. Variables persist across turns. observe expr emits hidden feedback for the next model step, and a block can do setup work without ending the user-visible turn.

Termination

finish expr always ends the turn, even when the rendered text is empty. observe expr never ends the turn. In headless mode, the task must eventually terminate through finish ... rather than prose alone.

Tool Access

In REPL mode, tools are called with call tool_name { ... }. Use call search_tools {} only when the needed tool is not already listed in the prompt. There is no T namespace or Python async layer.

Error Recovery

Parse errors include line/column context and runtime errors stay short and source-oriented. The intended recovery loop is: shrink the block, inspect with observe, then retry a narrower validated step.

05 Native Tools Provider-native tool call execution with structured results.
Execution Contract
Native Tools Execution Model
The Rust agent loop sends tool schemas to the provider. The model responds with native tool calls when execution is needed — no Python, no <repl>. Tool results are fed back as structured content for the next turn. Native-tools-only builds compile just this backend and make it the default.

Contract Rules

Use native tool calls when execution is needed. Do not emit <repl> blocks or Python code. When several tool calls are independent, emit them together in the same response. Use serial calls only when later arguments depend on earlier results.

Termination

Finish by returning a final assistant answer. No fake tool calls or placeholder arguments. Do not stop after tool execution unless the task is actually complete. In headless mode, keep calling tools until the task is done, then return a final answer.

Tool Access

The runtime exposes the available tools as native tool call schemas. Use only the tools shown in Available Tools. Prefer parallel independent calls in one turn when the arguments are already known. Prefer batch when you already know the arguments for several unrelated reads, searches, shell commands, or diagnostics and want them started together. Never invent tool names or arguments.

Parallelism

Good candidates for one-turn parallel calls: reading several files, multiple searches, independent inspections. Do not parallelize dependent steps, ordered stateful edits, or operations whose arguments depend on prior output.

No Persistent State

Unlike REPL mode, there is no Python namespace. Each tool call is stateless. Context comes from the transcript and prior tool results only.

Provider Streaming

Native mode streams structured tool-call parts from the provider layer. Grouped multi-call turns are replayed back to providers where the API supports it.

06 Tools & Delegation Composite tool graph, dynamic providers, and sub-agent delegation.
Tools
Composite Tool Graph
Static providers are wrapped by DynamicToolProvider (in-process adapter), then projected by dynamic capability profiles. CLI can stage edits via /tools and /caps, with apply on next turn.

Core Tools

read_file, apply_patch, exec_command, write_stdin, shell, shell_wait, shell_read, shell_write, shell_kill, glob, grep, ls, update_plan.

Reconfigure

All-or-nothing apply with generation checks; failures roll back and block turn dispatch.

Discovery

search_tools is for genuine discovery of tools not injected into the prompt. With no query, it returns the full active catalog.

Delegation
Sub-Agent Orchestration
agent_call forks parent dynamic state per sub-agent. Sub-agents share the same filesystem/worktree but get isolated tool/capability state. Low tier sub-agents are filtered (no write/delegation tools) and always execute in standard; medium/high inherit the parent session's execution mode. REPL-specific setup only applies when the child actually runs in repl.

Tiers

Low (read-only lookup, forced standard), medium (general), high (full capability). Each tier has a filtered tool set.

Recall Pattern

Use a low-tier read-only sub-agent to summarize prior context when needed. Both modes can use search_history/search_mem tool calls for prior-turn recall.

Constraint

Parallel sub-agents should avoid editing the same files concurrently.

Persistence
Store, Logs, and Session State
SQLite-backed store tracks archived context, history, memory, and agent state for the session, while CLI keeps session logs and runtime state envelope that can be restored on resume.

Store

Session-specific DB under ~/.lash/sessions/*.db.

Envelope

AgentStateEnvelope: messages, iteration, token usage, sub-agent state, snapshot.

07 Capabilities Dynamic capability profiles that control which tools and prompt sections are active.
Capability System
Dynamic Capability Profiles
Each capability is a named group that bundles tools, prompt sections, and runtime-call bindings. Capabilities can be toggled at runtime via /caps. All are enabled by default. Sub-agents inherit a fork of the parent's capability state.

Structure

Each CapabilityDefinition has an id, name, description, optional prompt section, runtime bindings, and a list of tool names it controls.

Projection

Only tools belonging to enabled capabilities appear in the active tool registry. Prompt sections from enabled capabilities are injected into the system prompt automatically.

Capability Registry
Built-in Capabilities
Nine capability groups ship by default. Each controls a coherent set of tools and optional prompt guidance.

Core Read

read_file, glob, grep, ls, search_tools. File navigation and discovery.

Core Write

apply_patch. File mutation tool.

Shell

exec_command and write_stdin in standard mode, plus shell, shell_wait, shell_read, shell_write, shell_kill in REPL mode.

Planning

update_plan plus ask in interactive standard sessions. In-memory checklist tracking and targeted user clarification.

Delegation

agent_call, agent_result, agent_kill. Sub-agent orchestration.

History

search_history. Persistent turn history search across prior completed turns.

Memory

search_mem, mem_set, mem_get, mem_delete, mem_all. Key-value memory.

Skills

load_skill, read_skill_file, search_skills. Skill discovery and loading.

Web

search_web, fetch_url. Web search and page fetch.

08 Memory & History Persistent key-value memory and turn history that survive context pruning.
Memory
Persistent Key-Value Memory
Memory is persistent key-value storage across context pruning. Use it to store durable decisions, conventions, and project-specific knowledge. In REPL mode, call the memory tools with call ... { ... }; in standard mode they are native tool calls.

Tool API

call mem_set { key, description, value } — store/update in REPL
call mem_get { key } — retrieve or null in REPL
call mem_delete { key } — remove in REPL
call search_mem { query, mode, regex, limit, keys } — search in REPL

Availability

search_mem is available in both execution modes, but REPL calls it as call search_mem { ... }.

Example — Storing a convention (REPL)
<repl>
memory_write = call mem_set { key: "repo_convention", description: "Use snake_case for tool names" }
observe memory_write
</repl>
Example — Searching memory (REPL)
<repl>
results = call search_mem { query: "naming convention" }
observe results
</repl>
Example — Searching memory (Native Tools)
assistant:
tool_call: search_mem
  query: "naming convention"
  limit: 5
History
Persistent Turn History
History search covers prior completed turns persisted by the runtime, including turns no longer visible in the active prompt window. Sub-agents can inherit parent-agent context at spawn time.

API

search_history(query, mode="hybrid", regex=None, limit=10, fields=None, since_turn=None) — search prior completed turns

Sub-Agent Inheritance

History and memory state can include parent-agent context inherited at spawn time. A low-tier sub-agent can call search_history and search_mem to summarize relevant context without full write access.

Example — Recalling prior context (REPL)
<repl>
# Find what the user asked about earlier
hits = call search_history { query: "database migration", limit: 5 }
observe hits
</repl>
Example — Delegating history recall (REPL)
<repl>
worker = call agent_call { prompt: "Search history for any database schema decisions and summarize them.", intelligence: "low" }
observe worker
</repl>
Use search_history and search_mem only when prior-turn recall is actually needed. Skip history-mining detours unless the task explicitly depends on prior turns.
09 Skills Discoverable, loadable skill packages that extend agent behavior at runtime.
Skills System
Skill Discovery & Loading
Skills are directory-based packages containing a SKILL.md manifest with description and instructions, plus optional supporting files. They live in ~/.lash/skills/ (global) and .agents/lash/skills/ (project-local), with project-local overriding global by name. The legacy .lash/skills/ path is still loaded as a fallback.

Structure

Each skill is a directory with a SKILL.md file. The manifest contains a description and instructions that get injected into the agent's context when loaded. Optional files alongside the manifest are accessible via read_skill_file.

Discovery

skills lists all available skills with names and descriptions. search_skills(query, ...) performs hybrid search across skill metadata. Both global and project-local skill directories are scanned.

Loading

load_skill loads the named skill and returns its instructions plus bundled file listing. The agent then follows the skill's instructions to complete the requested workflow.

Example — Discovering skills (REPL)
<repl>
results = call search_skills { query: "ui component library", limit: 5 }
observe results
</repl>
Example — Loading and using a skill (REPL)
<repl>
skill = call load_skill { name: "frontend-design" }
observe skill
</repl>
Example — Searching skills (Native Tools)
assistant:
tool_call: search_skills
  query: "UI component library"
  limit: 5
Skill Manifest
SKILL.md Format
The manifest is a markdown file parsed for name (directory name), description (first paragraph), and instructions (full content). Additional files in the skill directory are listed and accessible.

Resolution Order

~/.lash/skills/ is scanned first (global), then the repo-local paths with .agents/lash/skills/ preferred over legacy .lash/skills/. Project-local skills override global skills with the same name.

File Access

read_skill_file(name, path) reads supporting files bundled with the skill, such as templates, schemas, or reference code.

10 Dataflow Primary execution paths from host input through the agent loop to streamed output.

Shared Turn Lifecycle

01 Host Input

Frontend emits ordered TurnInput.items + optional image blobs.

02 Normalize + Apply

Runtime validates refs and applies pending dynamic reconfigure before turn start.

03 Agent Step

Prompt assembled from a mostly static prefix plus later dynamic sections for environment, memory, and tools, using the current execution mode.

04 Mode Exec

Execution diverges here: persistent REPL execution for repl, provider-native tool loop for standard.

05 Fold + Output

When prompt usage reaches the hard watermark, old turns are folded back to the soft watermark with one stable archive marker; events stream back and the final answer + updated envelope return as one canonical AssembledTurn.
