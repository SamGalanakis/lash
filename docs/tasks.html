How To Use This Page

Use this as a routing table. App-facing guides say what to wire; architecture pages explain why that wiring works.

Note

If two pages mention the same concept, the task row points at the canonical home. For example, session persistence lives in Persistence; replay and effect durability live in Durability and replay.

Turn Work

Running turns is the core application path. Pick collected output for backend jobs, streaming output for interactive UI, and queued turns for background/session-driven ingress.

run a turn

Build LashCore, open a LashSession, create a turn EffectScope, call session.turn(TurnInput::text(...)).run(scope).await?, then read TurnOutput::assistant_message(). Start with Quickstart, then use API basics for the full facade shape.

stream a turn

Implement TurnActivitySink and call TurnBuilder::stream(&sink, scope), collect_with(&sink, scope), or into_stream(scope). Fold TurnActivity.correlation_id into stable UI rows. See Turns and streams.

persist a session

Install a SessionStoreFactory on LashCoreBuilder::store_factory(...), use file-backed attachment/artifact stores, and reopen the same session id after restart. See Persistence wiring.

delete a session

Call LashCore::delete_session(..., scope) for factory-backed sessions. If a process registry is installed, Lash cancels and revokes session-owned process state before deleting the store state. See Delete sessions.

Host Surface

The host decides what the model can see and do: tools, prompt contributions, projected state, and plugin-specific input.

add a tool

Implement ToolProvider or use StaticToolProvider; advertise cheap manifests, resolve full contracts, and execute work in execute(). Register the provider through a plugin or core tool surface. See Plugin basics and Tool plugins.

expose state to RLM

Use RlmProjectedBindings for read-only names in Lashlang scope. Use JSON/value bindings for small state and ProjectionRef bindings for large host objects behind a resolver. See Projected bindings and RLM variables.

Long-Running Work

Use child sessions for delegated reasoning, Lashlang processes for reusable background work, and queued turns when background events should re-enter the model.

spawn subagents

Install SubagentsPluginFactory, choose or define capabilities, and let RLM call spawn_agent. Child sessions receive constrained specs and their usage rolls up to the parent. See Subagents.

start background work

Install a ProcessRegistry with .process_registry(...) plus a SessionStoreFactory so the default inline process worker can rebuild session runtimes, or pass .process_work_driver(...) when an external durable runner owns work execution. Then observe or cancel runs through session.process_control(). See Process work and Durability and replay.

Inspect And Debug

Use traces for runtime-level evidence and read views for committed app-facing state.

inspect traces

Attach JsonlTraceSink with .trace_sink(Some(...)), set TraceLevel, and render the JSONL with lash-trace-viewer. Add a separate Lashlang execution sink when you need foreground/process execution graphs. See Tracing.
