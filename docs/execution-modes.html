execution/modes

A mode decides one thing: how the model acts inside a turn. Standard mode issues native provider tool calls; RLM has the model write Lashlang programs. Everything else (the session, the durable per-turn commit, your host facets) is identical. Pick the mode that fits the turn, not a different runtime.

Two Modes, One Commit

Both modes open the same LashSession handles and settle each turn as one atomic RuntimeCommit against the session graph. The mode only changes the shape of the model's action, not the durability boundary or the host surface it reaches.

standard

The model emits provider-native tool calls plus prose. Lash dispatches the calls (concurrently when more than one is requested) and feeds results back until the turn settles. Smaller prompt, fewer protocol rules.

rlm

The model writes one fenced lashlang program per iteration, executed in a sandboxed VM where every effect crosses the host boundary. One turn can compose state reads, tools, loops, validation, subagents, and background work.

locked per session

A session's mode is fixed when it opens. Switching modes means opening a new session; the runtime never silently swaps protocols mid-session.

Standard Mode

Standard is the default action shape and the smaller surface. The model reasons in prose and requests provider-native tool calls; Lash executes them and continues the turn. Reach for it when one tool call plus prose is enough per step.

model action

Provider-native tool calls, exactly as the model's API exposes them. No language to learn, no VM in the loop.

concurrency

When the model requests several tool calls at once, Lash dispatches them concurrently and joins the results before the next model step.

cost

The smallest prompt and the fewest protocol rules: the right default for chat-shaped and single-step tool turns.

use lash::LashCore;

// `standard()` selects native provider tool-calling, the default mode.
let core = StandardCore::builder()
    .provider(provider)
    .model(model)
    // ... in-memory or durable runtime facets ...
    .build()?;

// A plain `open()` runs the session in standard mode.
let session = core.session("chat-1").open().await?;

For the full core/provider/facet setup this elides, start from Quickstart, which builds a standard-mode core end to end.

RLM Mode

RLM (the reasoning-language model protocol) hands the model a typed scratchpad instead of one tool call at a time. It writes Lashlang programs that loop, branch, validate, call tools, and submit a structured result, all inside a single durable turn.

model action

One closed ```lashlang block per iteration, executed in the active AgentFrame VM. Lash records reasoning, code, print output, diagnostics, images, and terminal values.

host boundary

Lashlang has no direct filesystem, process, network, provider, or database access. Every useful operation crosses the linked host surface and the runtime effect controller.

composition

State reads, repeated host calls, control flow, subagents, and background processes compose inside one turn that still commits exactly once.

// Build an RLM core for Lashlang-driven turns.
let core = lash::RlmCore::builder()
    .plugins(lash::plugins::runtime_plugin_stack())
    .provider(provider)
    .model(model)
    // ... runtime facets ...
    .build()?;

let session = core.session("task-1").open().await?;

The RLM protocol page covers prompt construction, the execution loop, submit semantics, and failure modes; the Lashlang language reference covers the syntax the model writes.

Choosing a Mode

The decision is per turn-shape, not per app; many hosts run both, one session each. When in doubt, start standard and move a turn to RLM once it needs to compose work the model can't express in a single tool call.

Choose RLM

The model needs a typed scratchpad, repeated host calls, loops or validation, structured submit, projected state, or Lashlang background/process abilities.

Choose standard

One provider-native tool call plus prose is enough. No language for the model to author.

Not sure which fits? Try both against a real task in the lash CLI. It runs either mode from the terminal, no embedding required, so you can compare them on real work before wiring one into your app.
