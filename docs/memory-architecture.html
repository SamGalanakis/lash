Rethinking long-running sessions

Keep the prompt small. Make memory a tool.

Instead of dragging full raw history through every turn, keep only a small recent window in active prompt, store older turns as durable memory plus artifact-backed tool output, and let the model pull older context on demand through a memory retrieval tool that returns compact, source-backed summaries.

Recent window
Last N turns

Only the freshest conversational state stays raw in active context.

Memory retrieval
get_memory

Query older decisions, plans, files, and tool evidence only when needed.

Artifacts
Full outputs

Large tool results live out of band, referenced by compact previews and stable IDs.

What changes

This shifts memory from “prompt stuffing” to “retrieval plus provenance”.

Prompt size stops scaling with session age Old turns age out of the active window without being lost.
Cache stays healthier History is mostly append-only; retrieved memory lands as fresh suffix rather than rewriting the prefix.
Tool output becomes inspectable without bloating context The model sees compact previews, while humans and follow-up tools can dereference full artifacts.
Core stays smaller if plugins own policy Tools and plugins decide how to summarize, retain, and retrieve their own evidence.
Design position

History folding becomes a last-resort safety valve. The primary path is recent raw turns + retrievable memory + artifact references.

02 / Tradeoffs

Why this is a better primary strategy than smarter folding alone

Folding and placeholder rewrites still have value, but they are destructive mechanisms. A memory tool architecture keeps the active prompt intentionally small and turns older context into a retrieval problem instead of an ever-growing serialization problem.

Current bias

Archive or rewrite history after it gets too big

Useful as a safety valve, but it lets the prompt grow first and then pays a one-time rewrite cost when compaction or placeholders land.

  • Large raw tool output can accumulate before folding ever triggers.
  • Replacing earlier content with placeholders rewrites the prefix and can bust cache once.
  • The main agent still carries exploration churn until the archive step happens.
  • History policy tends to drift into core because it operates after generic rendering.
Proposed bias

Small recent window plus explicit memory retrieval

Make the default steady state cheap. Older context remains available, but only arrives when the model asks for it or when the runtime has a strong reason to fetch it.

  • Prompt cost scales with recent activity, not total session age.
  • Cache stays healthier because retrieval adds fresh suffix instead of rewriting older bytes.
  • Tool evidence can stay rich in artifacts without being rich in prompt tokens.
  • Retrieval policy can live in plugins and tools instead of turning core into a compaction policy engine.
03 / Extensibility

The plugin system can express this, but it needs stronger first-class surfaces

The important design move is to keep core responsible for durable primitives and lifecycle, while plugins and tools own summarization, retention, retrieval, and render policy. That preserves flexibility without forcing every strategy into hardcoded runtime branches.

What core should own

Durable primitives, not memory policy

Core should supply stable building blocks so plugins do not invent incompatible temp-file and rewrite conventions.

Artifacts Session-owned artifact API with provenance: session_id, message_id, part_id, kind, retention, and dereference.
Part metadata Per-part durable fields for history state: compacted, preview kind, artifact ref, retrieval hints.
Resolved history A clean render pipeline where history is assembled from structured parts before the final prompt shape is chosen.
Retrieval hook A core tool surface can invoke a plugin-owned retrieval pipeline without embedding a fixed search algorithm in runtime.
What plugins and tools should own

Tool-aware memory policy

Each tool or plugin knows best what a compact history record looks like. Generic truncation stays as fallback, not as the main strategy.

Projection exec_command keeps exit state and tail output, read_file keeps path and line window, grep keeps counts and top hits, batch keeps child summaries.
Retention Plugins decide when full output spills into artifacts and when old previews become slimmer.
Retrieval get_memory should be implemented by a plugin-owned memory service that can search summaries, artifacts, and old turn metadata.
Ranking Plugins can provide ranking, summarization, or helper-agent retrieval without changing the outer tool contract.
Gaps to close

Needed plugin upgrades

The current system is close, but some hooks are too coarse if you want this to feel natural instead of bolted on.

Part-level hooks Plugins should be able to rewrite or annotate history at the part level, not only by replacing whole messages.
Composable projectors History projection should allow layered behavior: ANSI scrubber, generic truncator, then tool-specific summarizer.
Artifact refs Artifact references should be first-class structured values, not ad hoc strings hidden in JSON payloads.
Memory service A plugin-registered retrieval service should be able to back tools like get_memory cleanly.

Extension model to aim for

Tool plugin Exposes output preview strategy, artifact spill policy, and retrieval hints.
History plugin Selects recent raw window, composes memory summaries, and controls retrieval prompts.
Memory service plugin Indexes turns and artifacts, answers get_memory queries, optionally with a cheap helper model.
Core runtime Stores artifacts, preserves provenance, and renders whichever prompt shape the execution mode wants.
04 / Rollout

A clean sequence that gets value early without hardcoding the wrong abstraction

Start by making tool output cheaper in history, then introduce artifacts, then promote retrieval into a first-class tool. That keeps the system moving toward the end state without committing to a brittle compaction-first architecture.

Phase 1

Tool-aware history projection

Replace generic string truncation with tool-specific BeforeHistory summaries and more aggressive shell cleanup.

  • Strip ANSI and progress noise.
  • Keep compact previews instead of full JSON wrappers.
  • Keep recent raw turns, but make each turn lighter.
Phase 2

Artifact-backed outputs

Introduce session-owned artifacts and let large tool outputs spill into them by policy, with preview plus provenance in history.

  • First-class artifact refs.
  • Per-part metadata for preview kind and retention.
  • UI and tools can dereference without bloating prompt history.
Phase 3

get_memory as retrieval tool

Give the main model an explicit memory lookup tool backed by summaries, file references, and artifact indexes.

  • Query one or more focused memory questions.
  • Return compact source-backed context.
  • Keep helper-agent logic hidden behind the tool.
Phase 4

Use folding only as fallback

Retain current archival/folding as an emergency mechanism, not as the normal way old context leaves the prompt.

  • Overflow still has an escape hatch.
  • Main path stays retrieval-first.
  • Cache remains healthier over long sessions.
Core idea Recent turns are context. Older turns are memory. Large evidence is an artifact.
Design constraint Core should expose durable primitives. Plugins should own summarization, retention, retrieval, and prompt policy.
