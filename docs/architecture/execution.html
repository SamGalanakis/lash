Mode Execution

Modes contribute protocol drivers and preambles. The runtime loop does not need provider-specific branches to understand Standard versus RLM behavior.

standard

Native provider tool calls

StandardDriver::handle_llm_success interprets provider output. Tool calls become StartTools, then RuntimeTurnDriver::run_tool_calls dispatches native mode tools first and then registered ToolProviders.

rlm

Persistent Lashlang execution

RlmDriver extracts a closed lashlang fence, starts ExecCode, and RlmModeSession::execute_code runs the program through RlmExecutionState and HostBridge. Control tools such as continue_as are mode tools; llm_query is supplied by lash-llm-tools.

background

Subagents and monitors

Subagents, monitor tasks, and async tool calls share handle-shaped background work. RLM discovers live handles with list_async_handles and resolves them with await.

Embedded API

lash-embed is the higher-level app-facing API used by the examples. It hides CLI bootstrap details while leaving providers, app persistence, HTTP protocols, auth, and frontend streaming under the host application's ownership.

LashCore

Cloneable shared core that owns the runtime environment, default session policy, installed mode presets, optional store factory, tool providers, plugin factories, attachment store, trace sink, residency, sanitizer, and termination policy.

ModePreset

Installs semantic modes such as ModePreset::standard() and ModePreset::rlm(). Hosts can install both and choose a default with default_mode, then override per session with .standard(), .rlm(), or .mode(...).

LashSession

Represents one app conversation. It wraps a mutexed LashRuntime, exposes run(Input), turn(Input), and read_view(), and can carry a parent session id for app-level parent/child relationships.

TurnBuilder

Lets apps attach a TurnEventSink, cancellation token, and per-turn mode options before running. The final TurnResult contains final text, outcome, messages, tool calls, token usage, and turn issues.

let core = LashCore::standard()
    .provider(provider)
    .model("anthropic/claude-sonnet-4.6")
    .max_context_tokens(200_000)
    .store_factory(store_factory)
    .build()?;

let session = core.session(chat_id).standard().open().await?;
let result = session
    .turn(Input::text(user_text))
    .events(&events)
    .run()
    .await?;

Example shape

examples/mock-chat-service keeps a map of LashSessions keyed by conversation id and records projected TurnEvents. examples/openrouter-chat-ui adds an app database, a SessionStoreFactory, Axum routes, and SSE-style browser streaming.

Boundary rule

The embed API is intentionally above LashRuntime but below an app framework. It should not import CLI/TUI vocabulary, own product chat storage, or dictate HTTP and frontend protocols.

Background Work

Subagents are sessions, not special messages. Monitors are background tasks that wake future turns from stdout lines. Generic async tool calls are tracked inside the session.

Host Rendering

The terminal receives both live stream events and completed authoritative read views.

Interactive TUI

run_app drives event handling. Completed turns reconcile with finish_turn_from_read_view; live markdown lanes and activity state bridge partial output until the final projection arrives.

Export and trace

lash-export reads store state plus the full provider trace JSONL when available, then renders chronological projections, prompt snapshots, context percentages, cached-token percentages, and usage bars. lash-trace-viewer reads JSONL traces and displays timeline, LLM calls, stream events, and raw records.
