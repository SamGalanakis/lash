Crate Families

These are the high-value ownership groups to inspect before changing behavior.

Pure protocol

lash-sansio

Owns TurnMachine, Effect, Response, ModePreamble, prompt templates, messages, attachments, and tool definitions/output contracts.

lash-sansio/src/{sansio,turn,mode,prompt,tool_surface}.rs
Async runtime

lash

Owns LashRuntime, Session, SessionGraph, and RuntimePersistence — plus tool dispatch, provider registry, MCP, OAuth, attachments, tracing, and runtime controls.

lash/src/runtime/*, lash/src/session.rs
Mode plugins

standard + rlm

Standard drives native provider tool calls. RLM extracts lashlang code fences, runs persistent REPL state, snapshots execution globals, and records RLM trajectory.

lash-mode-standard, lash-mode-rlm, lashlang
Provider plugins

OpenAI, Codex, Anthropic, Google

Each provider implements state, auth, readiness, transport, and model policy components behind ProviderHandle.

lash-provider-*, lash-providers-builtin
Tool plugins

default tools + subagents + discovery

Default tools, subagents, monitors, plan mode, prompt context, rolling history, observational memory, and UI activity all register through plugin factories. Tool discovery is itself split — catalog projection, definitions, provider execution, lexical ranking, LLM rerank, and schema indexing.

lash-default-tools, lash-subagents, lash-plugin-*
Host UI

CLI, TUI, UI surfaces

The CLI bootstraps runtime, plugins, and providers, then renders the authoritative SessionReadView through ChronologicalProjection into UiTimeline.

lash-cli, lash-ui, lash-tui
Inspection tools

trace + export + file index

lash-trace defines JSONL trace records. lash-export renders saved sessions. lash-file-index powers async fuzzy file completion.

lash-trace, lash-trace-viewer, lash-export, lash-file-index
Experimentation

autoresearch + GEPA + benches

Autoresearch adds hidden tools and UI surfaces. GEPA and benchmark runners sit at the edge for evaluation and performance work.

lash-autoresearch, lash-gepa, bench/*/runner

Provider crates depend on runtime contracts; mode and plugin crates register behavior; the CLI composes the concrete app.
