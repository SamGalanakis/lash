RLM Execution Path

The model emits code, the runtime executes it locally with the host's ProjectedBindings and resource handles in scope, and the resulting trajectory is persisted as protocol events. A block that invokes continue_as through an explicit receiver finishes the turn as TurnOutcome::AgentFrameSwitch instead of looping back to the model.

An end-to-end fanout-and-merge program for a host surface that enables lowercase resource operations and background processes. It enumerates candidate files, starts per-file analyses as handles, awaits them as a record, builds a histogram with nested path assignment, and either finishes the trajectory or hands the partial work to a new AgentFrame when the budget would be exceeded.

```lashlang
process review_file(agent: Agents, path: str) {
  report = await agent.spawn({
    task: format("review {} for TODOs and panics", path),
    capability: "explore",
    output: Type { owner: str, todos: int, panics: int }
  })?
  finish report
}

paths = await workspace.default.glob({ pattern: "crates/lash/src/**/*.rs" })?
if empty(paths) {
  submit { reviewed: 0, by_owner: {} }
}

// Fan out: one subagent handle per file, capped at the first eight.
batch = slice(paths, null, 8)
handles = {}
for path in batch {
  handles[path] = start review_file(agent: agents.default, path: path)
}

// Resolve the whole record concurrently, then collapse into a histogram.
resolved = await handles
by_owner = {}
errors = []
for path in keys(resolved) {
  wrapper = resolved[path]
  if !wrapper.ok {
    errors = push(errors, { path: path, error: wrapper.error })
    continue
  }
  report = wrapper.value
  current = by_owner[report.owner]
  by_owner[report.owner] = {
    todos: (current == null ? 0 : current.todos) + report.todos,
    panics: (current == null ? 0 : current.panics) + report.panics
  }
}

// Hand off the rest if we only got through a slice; otherwise submit.
if len(paths) > len(batch) {
  await control.continue_as({
    task: "continue the review over the remaining files",
    seed: { remaining: slice(paths, 8, null), by_owner: by_owner, errors: errors }
  })?
}
submit { reviewed: len(batch), by_owner: by_owner, errors: errors }
```

Surface Build And Dispatch

The runtime owns the concrete bridge between authored Lashlang and host capabilities. A turn builds a RuntimeExecutionContext from the active session, tool dispatch context, plugin session, process services, artifact store, attachment store, and turn metadata. That context produces the LashlangSurface, links the module, stores the content-addressed artifact, and gives HostBridge the dispatch primitives it needs during VM execution.

Lashlang operationRuntime path
await board.play({ ... })?ToolAgentSurface maps board.play to a host tool name such as play_move; HostBridge calls RuntimeExecutionContext::call_tool through the active ToolDispatchContext.
cron.Schedule({ ... })A plugin-provided ResourceCatalog entry lowers this to a host value with type metadata. It is pure VM data until code passes it to triggers.register.
await triggers.register({ source, target })?The trigger host operation is routed to PluginSession's session trigger registry. The registry validates the source event type against the target process input and stores a route containing module/process refs, not source code text.
start name(...)The compiled effect carries module_ref, process_ref, required_surface_ref, process name, and args. Runtime process start writes a ProcessInput::LashlangProcess row and returns a process handle.
sleep for "5s"HostBridge converts the duration and runs a typed sleep effect through the runtime effect controller, so durable controllers can replay or schedule it.

The linked artifact is the durable identity boundary. The process worker reloads ModuleArtifact by module_ref, verifies the recorded required_surface_ref, then compiles the selected process_ref through the process cache. Live callable tool dispatch still uses the current tool surface, so the artifact captures the language contract while the host remains responsible for active tools and resources.

Images And Attachments

Image support is a first-class runtime feature, not a plain record convention.

print(image)
Collects image values recursively, registers their attachment refs, and sends provider-visible LlmContentBlock::Image content alongside descriptor text.
submit(image)
Serializes metadata only, using a JSON-compatible descriptor with id, label, size, width, and height. It does not embed base64 data.

Runtime Internals

The implementation compiles AST to a compact VM instruction stream, then executes async host effects without blocking the RLM runtime.

  1. Parser

    The lexer tokenizes strings (including r"""...""" / r'''...''' raws), // comments, operators, and keywords. The parser builds a module AST rooted at Program, with declarations, assignment targets, receiver operations, named process starts, blocks, control flow, and type expressions. Host ability checks happen later in the linker, not in the grammar.

  2. Compiler

    The compiler lowers linked canonical IR into a Chunk of Instructions, folds static type literals, compiles assignment paths, loop cleanup jumps, builtin calls (specialized to Instruction::Intrinsic ops like IntrinsicOp::Len, Range, Push, FormatCompiled, ValidateCompiled, etc.), and explicit resource/process effects that the linked LashlangSurface allowed. Chunks no longer carry the full AST or LinkedModule; process-start effects carry module_ref, process_ref, required_surface_ref, display name, and args. The process runner loads the ModuleArtifact from the artifact store and uses a compiled-process cache keyed by (process_ref, required_surface_ref, compiler_version, vm_abi_version); live resource operation dispatch still uses the current callable tool surface.

  3. VM

    The VM walks the compiled Chunk, stores globals in State, dispatches host effects through ExecutionHost, recursively awaits records/lists of handles, and consults the ProjectedBindings for any name not present in State. It is split into instruction dispatch, the VmEffect seam that calls into ExecutionHost::perform, the run loop with mode entry points and the VmStep / VmOutcome step machinery, and the optimizer's loop IR. Every enabled effectful op — resource operation, start, await, cancel, print, process start, process event, and the program terminators submit / finish / fail — goes through perform; handlers may transform the value but the VM enforces divergence for the terminator ops. Separate modules own value coercions and builtin bodies, field/index reads and path assignment, the format(...) and json_parse/serialization helpers, and the public compile / execute surface re-exported from the crate root.

  4. Profiling

    The profiling examples and Scenario::ALL cover whole-language performance sweeps, including compiler and builtin hot paths. ProfileReport aggregates per-instruction and per-builtin counts/times.
