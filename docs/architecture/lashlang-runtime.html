RLM Execution Path

The model emits code, the runtime executes it locally with the host's ProjectedBindings and resource handles in scope, and the resulting trajectory is persisted as protocol events. A block that invokes continue_as through an explicit receiver finishes the turn as TurnOutcome::AgentFrameSwitch instead of looping back to the model.

An end-to-end fanout-and-merge program for a host surface that enables lowercase resource operations and background processes. It enumerates candidate files, starts per-file analyses as handles, awaits them as a record, builds a histogram with nested path assignment, and either finishes the trajectory or hands the partial work to a new AgentFrame when the budget would be exceeded.

<lashlang>
process review_file(agent: Agents, path: str) {
  report = await agent.spawn({
    task: format("review {} for TODOs and panics", path),
    capability: "explore",
    output: Type { owner: str, todos: int, panics: int }
  })?
  finish report
}

paths = await workspace.default.glob({ pattern: "crates/lash/src/**/*.rs" })?
if empty(paths) {
  submit { reviewed: 0, by_owner: {} }
}

// Fan out: one subagent handle per file, capped at the first eight.
batch = slice(paths, null, 8)
handles = {}
for path in batch {
  handles[path] = start review_file(agent: agents.default, path: path)
}

// Resolve the whole record concurrently, then collapse into a histogram.
resolved = await handles
by_owner = {}
errors = []
for path in keys(resolved) {
  wrapper = resolved[path]
  if !wrapper.ok {
    errors = push(errors, { path: path, error: wrapper.error })
    continue
  }
  report = wrapper.value
  current = by_owner[report.owner]
  by_owner[report.owner] = {
    todos: (current == null ? 0 : current.todos) + report.todos,
    panics: (current == null ? 0 : current.panics) + report.panics
  }
}

// Hand off the rest if we only got through a slice; otherwise submit.
if len(paths) > len(batch) {
  await control.continue_as({
    task: "continue the review over the remaining files",
    seed: { remaining: slice(paths, 8, null), by_owner: by_owner, errors: errors }
  })?
}
submit { reviewed: len(batch), by_owner: by_owner, errors: errors }
</lashlang>

Surface Build And Dispatch

lash-lashlang-runtime owns the concrete bridge between authored Lashlang and host capabilities. A turn builds a RuntimeExecutionContext from the active session, tool dispatch context, plugin session, process services, artifact store, attachment store, and turn metadata. The Lashlang runtime uses that context to produce the LashlangHostEnvironment, link the module, store the content-addressed artifact, and give HostBridge the dispatch primitives it needs during VM execution.

Lashlang operationRuntime path
await board.play({ ... })?LashlangToolBinding maps board.play to a host tool name such as play_move; HostBridge calls RuntimeExecutionContext::call_tool through the active ToolDispatchContext.
cron.Schedule({ ... })A plugin-provided LashlangHostCatalog entry lowers this to a host descriptor with type metadata. It is pure VM data until code passes it to triggers.register.
await triggers.register({ source, target, inputs })?The trigger host operation is routed to the runtime trigger store. Registration validates the source event type against every trigger.event input, computes the source key, and stores a durable subscription from source type/key to process definition plus a fixed/event input template.
start name(...)The compiled effect creates one process run from the named process definition and returns a run handle.
sleep for "5s"HostBridge converts the duration and runs a typed sleep effect through the runtime effect controller, so durable controllers can replay or schedule it.

The linked artifact is the durable identity boundary. Process start validates a captured execution environment before registration: Lash rebuilds a candidate plugin session and Tool Catalog from the process's Process Plugin Options, builds the candidate LashlangHostEnvironment, and checks it satisfies the artifact's Host Requirements. The process worker then reloads ModuleArtifact by module_ref, verifies the recorded host_requirements_ref, rebuilds the environment from the same captured options, checks the requirements again, and compiles the selected process_ref through the process cache. Live callable tool dispatch still uses the rebuilt current tool catalog for that process environment, so the artifact captures the language contract while installed plugins own how snapshot-backed tools and resources are reconstituted. Static @label process-map metadata is part of the artifact hash and recorded surface requirements; changing a title or description changes the module ref but not the compiled execution behavior.

Images And Attachments

Image support is a first-class runtime feature, not a plain record convention.

print(image)
Collects image values recursively, registers their attachment refs, and sends provider-visible LlmContentBlock::Image content alongside descriptor text.
submit(image)
Serializes metadata only, using a JSON-compatible descriptor with id, label, size, width, and height. It does not embed base64 data.

Runtime Internals

The implementation compiles AST to a compact VM instruction stream, then executes async host effects without blocking the RLM runtime.

  1. Parser

    The lexer tokenizes strings (including single quotes, double quotes, triple quotes, and raw r-prefixed quote forms), # and // line comments, operators, annotation sigils, and keywords. The parser builds a module AST rooted at Program, with declarations, assignment targets, receiver operations, named process starts, blocks, control flow, type expressions, and optional @label metadata on process declarations or process-map node statements. Host ability and language-feature checks happen later in the linker, not in the grammar.

  2. Compiler

    The compiler lowers linked canonical IR into a Chunk of Instructions, folds static type literals, compiles assignment paths, loop cleanup jumps, builtin calls (specialized to Instruction::Intrinsic ops like IntrinsicOp::Len, Range, Push, FormatCompiled, ValidateCompiled, etc.), and explicit resource/process effects that the linked LashlangHostEnvironment allowed. LabelAnnotated nodes are transparent to bytecode generation and preserved only for artifact identity and static process mapping. Chunks no longer carry the full AST or LinkedModule; process-start effects carry module_ref, process_ref, host_requirements_ref, display name, and args. The start path validates those refs against the captured process environment before row registration. The process runner loads the ModuleArtifact from the artifact store and uses a compiled-process cache keyed by (process_ref, host_requirements_ref, compiler_version, vm_abi_version); live resource operation dispatch uses the callable tool catalog rebuilt from the process's captured Process Plugin Options.

  3. Compilation caching

    Compilation is cached on two seams so repeated source and repeated processes skip the parse/link/lower pipeline. The foreground RLM executor holds a LinkedProgramCache on its RlmExecutionState; each paired <lashlang> block calls get_or_compile(code, surface), which returns a cached CompiledLinkedProgram (the LinkedModule plus its Arc<CompiledProgram>) when the source hash matches and the current LashlangHostEnvironment still satisfies the cached artifact's host_requirements; a miss parses, links, and compiles once, then stores it. A surface change that no longer satisfies the recorded requirement misses and recompiles, so cache reuse never weakens linker gating. The cache is a pure accelerator outside the snapshot, so it is rebuilt per session and never serialized. The durable side uses CompiledProcessCache, keyed by (process_ref, host_requirements_ref, compiler_version, vm_abi_version) so a process artifact compiles once per worker. Both caches are bounded LRU rings and report CompiledProgramCacheStats (hits, misses, evictions, entries, capacity).

  4. VM

    The VM walks the compiled Chunk, stores globals in State, dispatches host effects through ExecutionHost, recursively awaits records/lists of handles, and consults the ProjectedBindings for any name not present in State. It is split into instruction dispatch, the VmEffect seam that calls into ExecutionHost::perform, the run loop with mode entry points and the VmStep / VmOutcome step machinery, and the optimizer's loop IR. Every enabled effectful op (resource operation, start, await, cancel, print, process start, process event, and the program terminators submit / finish / fail) goes through perform; handlers may transform the value but the VM enforces divergence for the terminator ops. Separate modules own value coercions and builtin bodies, field/index reads and path assignment, the format(...) and json_parse/serialization helpers, and the public compile / execute surface re-exported from the crate root.

  5. Profiling

    The profiling examples and Scenario::ALL cover whole-language performance sweeps, including compiler and builtin hot paths. ProfileReport aggregates per-instruction and per-builtin counts/times.
