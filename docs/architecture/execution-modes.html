Standard

Standard mode is closest to ordinary tool-calling chat. The provider receives the rendered conversation plus native model tool specs.

What Goes In Context

  • Rendered conversation messages from MessageSequence.
  • Attachments extracted by prompt rendering.
  • System prompt from the active PromptTemplate.
  • Native provider tool schemas from ToolSurface::model_tool_specs().
  • No RLM trajectory field is rendered separately.

Execution Notes

StandardDriver consumes text, reasoning, and native tool-call parts. The runtime records tool invocations through TurnGraphOverlay as tools complete.

RLM

RLM still uses chat-shaped context, but tools are only callable from lashlang. The provider sees an empty native tool schema.

What Goes In Context

  • Rendered conversation history, including user and assistant messages.
  • The RLM execution prompt and shared lashlang language reference.
  • Available tool documentation as prompt text, not provider tool schemas.
  • RLM feedback messages when the model emits prose without required lashlang or when typed validation fails.
  • The active persistent REPL state is not serialized as native provider tools; the model accesses it through subsequent lashlang blocks.

Completion Contract

RLM can be configured with RlmTermination::Finish for submit-based typed/final output. Legacy root chat can use ProseWithoutFence in specific cases, but lashlang remains the execution path.

RLMPure

RLMPure is the compact trajectory variant. It does not send chat history wholesale; it formats task inputs and REPL trajectory explicitly.

What Goes In Context

  • Inputs: up to four user text/prose chunks, preserving the first and the most recent messages when history is long.
  • REPL history: up to twelve RlmTrajectoryEntry steps with reasoning, lashlang source, observation preview, error, and final output when present.
  • Iteration: one-indexed iteration number for the next prediction.
  • System prompt: RLMPure execution discipline plus the shared lashlang language reference.
  • No native tools: tools are prompt-documented and callable only through call tool_name { ... }.

Iteration Discipline

The prompt now explicitly says not to solve everything in one block and not to submit in the same block that calls a tool or prints. The intended loop is: write code, observe output, then decide the next code block or final submit.
