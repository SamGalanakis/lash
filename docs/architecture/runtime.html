runtime/host

The async runtime is the effect controller boundary around the pure turn machine. It manages residency, persistence, plugin hooks, provider calls, background work, usage accounting, and read projections while keeping CLI rendering and the app-facing lash API outside the core session model.

Runtime Shape

LashRuntime holds the live session registry, the managed-sessions map, the active handoff continuation table, the pending first-turn input queue, and a usage ledger. Sans-IO computes protocol effects, RuntimeEffectController handles nondeterministic host work, plugins contribute behavior, providers complete requests, and persistence records completed turns plus in-flight turn checkpoints and effect outcomes.

Runtime Effect Controller

RuntimeTurnDriver only polls TurnMachine, builds an RuntimeEffectEnvelope, invokes the active scoped RuntimeEffectController, and feeds the returned response back. The default inline controller executes local turn work; durable controllers run a turn through RuntimeEffectControllerScope with a stable turn id.

Every invocation carries session id, origin, turn index, mode iteration, effect id, effect kind, a stable idempotency key, ref-only request data, and a checkpoint digest for turn-originated effects. The checkpoint payload is stored as RuntimeTurnCheckpoint, while completed outcomes are stored as RuntimeEffectJournalRecord keyed by idempotency key and envelope hash. When persistence is configured, named turn effects require an active RuntimeTurnLease; the runtime renews that same token before checkpoint saves and before journal outcome saves, abandons ownership on non-commit exits, and lets final commit clear in-flight rows atomically. Cancellation, streams, provider trace sinks, and attachment byte hydration stay in the local executor or host workflow. The boundary covers turn LLM calls, direct/plugin LLM completions, individual tool calls, retry sleeps, RLM exec, checkpoints, and execution-surface sync.

RuntimeCoreConfig::with_effect_controller, RuntimeEnvironmentBuilder::with_effect_controller, and EmbeddedRuntimeBuilder::with_effect_controller configure the required runtime controller. Durable workflow adapters should prefer the scoped turn entrypoints so the borrowed workflow controller and stable turn identity are explicit for that run, then use LashRuntime::resume_turn or LashSession::resume_turn(turn_id) after a crash or worker move.

Residency And Sessions

Lash can run behind the CLI or inside any long-lived host application. The same session machinery supports active sessions, managed child sessions, parked runtimes, and resumable hosts.

interactive

CLI host

lash-cli composes providers, plugins, MCP servers, traces, and terminal rendering around a live LashRuntime.

application

Application sessions

lash exposes LashCore/LashSession for long-lived host applications. Hosts open sessions from ids and stores; parking/resume remains runtime plumbing behind the facade.

embedded

Host services

Services can compose the same core with their own providers, plugins, stores, traces, and validation around product-specific session ids.

Host Capabilities

RuntimeSessionManager implements one concrete RuntimeSessionHost boundary with typed operations for plugin and tool needs. Tool authors do not receive the runtime handle directly; ToolContext projects host capabilities into explicit methods such as session_model(), tool_catalog(), sessions(), processes(), and direct_completion(...).

Turn Outcomes And Handoffs

A turn finishes as one of three outcomes. App hosts usually call session.turn(input).stream(&sink), .run(), or .collect_with(&sink); those facade calls follow handoffs and return the final TurnResult. Raw stream_turn and stream_turn_following_handoffs are runtime internals for lower-level hosts.

OutcomeShapeMeaning
TurnOutcome::FinishedTurnFinish::AssistantMessage { text }, TurnFinish::SubmittedValue { value }, or TurnFinish::ToolValue { tool_name, value }The turn produced a final answer. AssistantMessage is the sole terminal prose result; live prose is only AssistantProseDelta, and the runtime commit materializes the settled transcript message once. SubmittedValue and ToolValue are typed terminal values authored by RLM submit or terminal tool controls.
TurnOutcome::Handoff{ session_id }RLM continue_as opened a successor session with a fresh window. The runtime queues the packed { task, seed } as that session's first turn input.
TurnOutcome::StoppedTurnStop::{Cancelled, InvalidInput, MaxTurns, ToolFailure, ProviderError, PluginAbort, RuntimeError, SubmittedError{...}, ToolError{...}}The turn ended without a clean answer. SubmittedError and ToolError carry terminal errors authored by failing control paths, such as submit_error, so hosts can surface them.

Tool Result Projection

The built-in projector enforces budgeted views of tool output across three hook sites so the durable graph, the live model prompt, and rolled-up history each see consistent, bounded content.

Persistence And Usage

Persistence is turn-scoped. Commits update the session graph, checkpoint blobs, session heads, usage ledger, and settled assistant prose together so read views and host reports agree. Modes describe outcomes; the shared runtime commit path handles final transcript materialization.
