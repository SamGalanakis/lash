Where To Start

Five crates cover the user-facing surface. Pick by role, not by stack position — most embedders only ever need lash-runtime.

lash-runtime ↗

The facade. LashCore, LashSession, TurnInput, SessionSpec, PluginStack. Imported as use lash::... (the crates.io name is lash-runtime because lash is owned by another project). Start here if you're embedding lash in an app.

lash-core ↗

The runtime kernel — EffectHost / RuntimeEffectController, plugin host, session graph, protocol build input, tool execution, process-registry store plumbing, Lashlang host environment construction, environment wiring. Drop down here when you're writing a custom PluginHost, explicitly installing a deployment-level lash::process::ProcessRegistry such as lash-sqlite-store::SqliteProcessRegistry or lash-postgres-store::PostgresProcessRegistry, or doing something the facade doesn't expose.

lash-sansio ↗

Pure protocol types — LlmRequest, LlmResponse, ToolDefinition, TurnDriverPreamble, the sans-IO state machine. No async, no IO, no Lashlang dependency. Reach for it when writing a provider or anything that touches the wire shape directly.

lash-remote-protocol ↗

Versioned DTOs for exposing Lash behind service boundaries: RemoteTurnRequest, RemoteTurnResult, RemoteLlmRequest, RemoteLlmResponse, prompt layers, activity streams, trigger occurrence/subscription envelopes, remote tool grants, and optional lash-core conversions. See the remote protocol guide for the contract.

lashlang ↗

The compact CodeAct language the model writes against under RLM protocol. Compiler, runtime, effects. Mostly internal — read it when extending RLM or porting Lashlang to a new effect surface.

The Rest Of The Workspace

The workspace contains 42 members. Most are integrations the facade composes for you (providers, plugins, tools, transports, stores, examples, and E2E harnesses) and you don't import them directly — your LashCore wiring picks them up.

  • Provider crates: lash-provider-anthropic, lash-provider-openai (which also ships the codex OAuth provider and OpenAI JSON Schema projection), lash-provider-google, and shared auth/transport support in lash-provider-auth and lash-llm-transport. The CLI wires provider factories for the compiled-in provider kinds directly.
  • Plugin crates: lash-plugin-mcp, lash-plugin-process-controls, lash-plugin-tool-discovery, lash-plugin-observational-memory, lash-plugin-plan-mode, and the lash-standard-plugins aggregator (which also bundles the rolling-history plugin).
  • Tool crates: lash-tools (shell, files, search, web, apply-patch modules), shared scaffolding in lash-tool-support and lash-llm-tools.
  • Mode crates: lash-protocol-standard, lash-protocol-rlm.
  • Persistence, tracing, durable execution, profiling, and remote DTOs: lash-sqlite-store, lash-postgres-store, lash-s3-store, lash-restate, lash-trace, lash-trace-viewer, lash-export, lash-perf, lash-remote-protocol.
  • Hosts: lash-cli, lash-tui, lash-tui-extensions.

Every published crate has its own page on docs.rs at https://docs.rs/<crate-name>. The crates.io index is at crates.io/search?q=lash ↗.
