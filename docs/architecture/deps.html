deps/edges

Lash keeps external systems at explicit edges: LLM transports, OAuth flows, MCP servers (stdio / streamable-http / sse), Tavily search/extract, models.dev metadata, trace sinks, file watchers, remote service DTOs, and pluggable runtime stores.

Providers

Three first-party provider crates ship five provider kinds: lash-provider-openai (direct openai Responses, openai-compatible Chat Completions, and the codex OAuth device-code flow), lash-provider-anthropic (anthropic), and lash-provider-google (google_oauth). They share one runtime interface (ProviderHandle) but own their own auth, readiness, model policy, transport, and cache-marker placement.

See the Providers chapter for the OpenAI Responses vs Chat split, Anthropic cache_control placement rules, RLM prompt-cache strategy, normalized LlmUsage shape, and the components needed to add a new provider.

Local And Optional Dependencies

These dependencies are important operationally because they affect startup, discovery, observability, or Tool Catalog membership.

MCP

lash-plugin-mcp wraps the official rmcp SDK and registers an McpPluginFactory with the lash plugin system. The factory keeps a shared per-LashCore connection pool, so stdio servers spawn once and every session built from the same core reuses the same client. McpServerConfig supports three transports: stdio (child process), streamable_http, and sse. Tools are surfaced under mcp__<server>__<tool> names, preserving input and output schemas.

Tavily

Web search and URL extraction register only when LashConfig.auxiliary_secrets.tavily_api_key is present.

models.dev

lash-cli owns models.dev parsing, refresh, and file caching, then passes a resolved ModelSpec into core.

Tracing

lash-trace defines schema version 2 JSONL records. Standard records cover turns, LLM starts/completions, tools, and summaries; extended records preserve provider/stream details. Lashlang execution graphs are an opt-in record stream for foreground blocks, durable process runs, node activity, branches, and child links. The optional otel cargo feature pulls in opentelemetry 0.31 and maps runtime trace events to spans. lash-trace-viewer is a standalone binary crate that consumes lash-trace JSONL directly, independent of lash-export.

Remote protocol

lash-remote-protocol defines versioned DTOs for service boundaries: remote turns, provider LLM calls, prompt layers, activity streams, trigger occurrence/subscription envelopes, and remote tool grants. Tool execution stays host-owned through normal ToolProvider implementations. The facade reexports the DTOs as lash::remote; the standalone crate's core-conversions feature adds adapters for hosts that already depend on lash-core. See Remote protocol.

File index

lash-file-index combines ignore walking, notify watching, and nucleo matching for async @ completion.

Store

lash-core defines runtime store contracts. First-party adapters cover local SQLite sessions and shared-worker Postgres state for session execution leases, pending turn input, queued work, processes, triggers, artifacts, blobs, usage, and garbage collection; hosts can implement the same traits on another database when they preserve the same fenced commit and claim semantics.

Export

lash-export renders persisted session state plus provider trace JSONL when present, so transcript views can include prompt snapshots, context percentage, cache-read percentage, cache-write counts, and exact token counts.
