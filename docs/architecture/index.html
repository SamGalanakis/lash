Current workspace architecture

lash/system

lash is a Rust workspace for an embeddable terminal coding agent. The current shape is a layered runtime: lash-sansio owns deterministic turn protocol and prompt assembly, lash hosts runtime, plugins, stores, providers, and sessions, and lash-cli assembles the default TUI application.

This folder is the architecture entrypoint. It replaces the old single-page architecture document with focused pages for modules/runtime, execution modes, lashlang, and file/dependency reference.

Core Truths

The architecture should be read from the repo-owned boundary outward: first the typed session graph and turn protocol, then host UX and provider details.

Prompt construction is mode-aware

build_turn combines the active ModePreamble, current ToolSurface, extra prompt contributions, and PromptTemplate. The resulting system prompt belongs to the turn machine config.

Context projection is separate from protocol handling

ContextProjector decides what messages and tool schemas the provider sees. ProtocolDriverHandle decides how model output advances the turn.

Session graph is the durable timeline

Conversation records, tool invocations, mode events, state snapshots, plugin nodes, and active path metadata are persisted as graph nodes.

RLM tool effects are now causal

Lashlang tool calls are recorded before the RLM trajectory/final assistant event, so durable order matches execution order: user, tool, trajectory, assistant.
