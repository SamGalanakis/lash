§ 01

the primitives

one runtime, the whole stack

An agent is a loop around a model, but a useful one needs more: durable state, code execution, subagents, background work. Assemble those from a workflow engine, a sandbox, and an orchestrator and you run several systems that never share a boundary.

lash is that whole stack in one runtime. Providers feed the turn loop, every effect crosses one typed boundary, the turn commits atomically, and the host composes the rest as plugins.

§ 02

plugins

compose the runtime

The runtime is assembled from plugins. Tools, prompts, planning, memory, history transforms, UI activity, and tool-output budgeting hang off about eighteen typed hooks; a host composes only what it embeds.

The execution mode is a plugin too. standard and rlm ship as separate protocol crates and install like anything else, so the loop's protocol is swappable.

Providers (Anthropic, OpenAI, Gemini, any OpenAI-compatible endpoint, and MCP servers) and tracing sinks (JSONL, OpenTelemetry) attach the same way.

§ 03

rlm

the model writes programs

In rlm mode the model's action is a whole program, not one tool call. It writes lashlang, a typed DSL the runtime runs deterministically. This is the CodeAct pattern, except every call crosses a host-granted ability, so the language has no IO of its own. A small controlled language, rather than Python, is a deliberate design choice.

// the model emits this; the runtime runs it deterministically.
// every call is a host-granted ability, the VM has no IO of its own.
hits = await web.search({ query: "rust async runtimes" })?

notes = []
for h in hits.results {
  page = await files.read({ path: h.path })?   // large output, read lazily
  notes = notes + [page.summary]
}

submit notes

Loops, branches, and many tool calls settle in a single turn. Large outputs bind lazily, so reading one field never pulls the whole blob into context.

The same language spawns subagents, picks an identity, and runs durable background work:

// a subagent with its own task and a typed result
finding = await agents.spawn({ task: "audit auth.rs", output: Finding })?

// identity is the path the model names: gmail.work, never gmail.personal
process notify(mail: Gmail) { await mail.send({ body: finding })? finish true }
start notify(mail: gmail.work)

agents.spawn runs a typed subagent on the side. Identity is the resource path the model names, and the host decides which paths exist. start launches a durable background process the turn can await, signal, or cancel.

§ 04

durability

durable by responsibility

Durability is split into two separate contracts: storing the completed turn, and replaying the individual effects inside a turn that crashes.

completed turns
The turn is the unit a host observes. It settles as one atomic commit: the session graph, head revision, and usage applied together. How the turn was executed stays out of this contract, so the settled state is coherent no matter what ran underneath.
in-flight effects
A durable turn provider handles the narrower question: what if the host crashes while one nondeterministic effect is in flight? Each effect (an LLM call, a tool call, process control, a retry sleep) is checkpointed and journaled, so the turn resumes and that effect runs exactly once.

Where that in-flight replay lives is host wiring: embedded, with lash journaling to a local SQLite store, or substrate-native, where a workflow engine like Restate owns replay instead. The turn-commit contract underneath is the same either way.

§ 05

embed it

embed it

The shortest path to a working turn: add the crate, pick a provider, open a session, run one turn. Tools, modes, persistence, and tracing layer on after that.

The facade ships on crates.io as lash-runtime and is imported as use lash::…. During the alpha series the versions carry an -alpha.N suffix, so the dep needs the explicit pre-release tag.

[dependencies]
lash-runtime         = "=0.1.0-alpha.16"
lash-provider-openai = "=0.1.0-alpha.16"
anyhow               = "1"
tokio                = { version = "1", features = ["full"] }

One LashCore per app, cloned freely; one LashSession per chat or task; .run() for a single collected turn, .stream(&sink) for live events.

use lash::{provider::ProviderHandle, LashCore, ModelSpec, TurnInput};
use lash_provider_openai::{OPENROUTER_BASE_URL, OpenAiCompatibleProvider};

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let api_key = std::env::var("OPENROUTER_API_KEY")?;
    let provider = ProviderHandle::new(
        OpenAiCompatibleProvider::new(api_key, OPENROUTER_BASE_URL).into_components(),
    );

    let model = ModelSpec::from_token_limits(
        "anthropic/claude-sonnet-4.6", None, 200_000, None, None,
    )
    .map_err(anyhow::Error::msg)?;

    // one LashCore per app, cloned freely.
    let core = LashCore::standard()
        .provider(provider)
        .model(model)
        .build()?;

    // one session per chat / task; run one turn; read settled prose.
    let session = core.session("hello-1").open().await?;
    let result = session
        .turn(TurnInput::text("Say hi in one short sentence."))
        .run()
        .await?;

    println!("{}", result.assistant_message().unwrap_or_default());
    Ok(())
}

§ 06

operations

running it

Embedded in a backend, the runtime gives you the operational surface a real service needs.

streaming
.run() collects a whole turn; .stream(&sink) and .into_stream() hand you activity events as they happen, so a UI can follow tokens, tool calls, and steps live.
resume
session.resume_turn(id) reloads an interrupted turn from its last checkpoint and finishes it: the durability layer, on the embedding side.
queued work
session.next_queued_turn() pulls the next turn from an app-owned queue with at-least-once or at-most-once delivery, so webhooks and task queues can drive turns asynchronously.
per-turn model
turn.provider(…) and turn.model(…) override the provider and model for a single turn, for fallbacks, cost routing, or multi-model dispatch.
