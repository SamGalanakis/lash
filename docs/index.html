Most agent stacks treat the LLM as the runtime and stitch state around it — a database for memory, a queue for retries, a sandbox for code. lash inverts that. § 01

the inversion

the runtime is the durable end

An LLM call is nondeterministic and can fail halfway. Treating it as the runtime means rebuilding durability around every model, every provider, every app. lash moves the durable boundary to the side that can actually hold it.

the usual shape

The model drives. Memory is a database you bolt on, retries are a queue you wire up, code execution is a sandbox you babysit. State lives in whatever the model didn't lose.

lash

The runtime drives. It owns the turn — model calls, modes, tools, plugins, semantic stream events, usage, terminal outcomes. Every completed turn lands as one durable commit. The model is the variable call inside it.

The split is the whole idea, and it is drawn on purpose:

your app owns
Storage, auth, transport, and product state — the outer boundaries. lash persists only what it needs to resume a turn and rebinds your live handles when a session opens.
lash owns
The turn: provider protocol, execution mode, tool dispatch, plugins, the semantic event stream, usage accounting, and the single lease-fenced commit that settles it.

§ 02

what's inside

what's inside

Seven pieces do the work. Each is a real seam in the codebase, not a slogan — follow any of them into the architecture chapters.

durable

per-turn commits

Every completed turn lands as one lease-fenced RuntimeCommit against a SessionGraph — graph delta, checkpoint blobs, usage deltas, and head revision in one SQLite transaction with optimistic CAS. In-flight turns persist separately and replay on resume.

sans-io

effect controller

The RuntimeEffectController is the durable boundary around nondeterministic work. Model calls, tool calls, RLM exec, process control, and checkpoints all cross it as a typed RuntimeInvocation — so a turn can be replayed from the journal exactly once.

modes

two execution modes

standard uses the provider's native tool-calling protocol and dispatches the model's parallel tool calls concurrently. rlm runs lashlang programs in a sandboxed VM. Both settle into the same single commit unit.

lashlang

a typed DSL the model emits

A small language the model writes and the runtime executes deterministically. Host capabilities are lowercase module ops — web.search, files.read, agents.spawn. Unavailable abilities still parse but fail at link time and never reach the prompt.

plugins

everything is a plugin

Tools, prompts, planning, UI activity, subagents, memory, history transforms, and tool-output budgeting are all plugins. Host applications compose only what they need through the lash facade.

providers

provider portability

First-party crates for Anthropic, OpenAI Responses, any OpenAI-compatible Chat Completions endpoint, OpenAI Codex, and Google Gemini / Code Assist. MCP servers attach over stdio, streamable-HTTP, or SSE.

tracing

tracing as a sink

JSONL by default with a self-contained HTML viewer for reading a turn back event by event. OpenTelemetry export is one line when you want it.

§ 03

embed it

embed it

The shortest path to a working turn: add the crate, pick a provider, open a session, run one turn. Tools, modes, persistence, and tracing layer on after that.

The facade ships on crates.io as lash-runtime (the bare name is owned by another project) and is still imported as use lash::…. During the alpha series the versions carry an -alpha.N suffix, so the dep needs the explicit pre-release tag.

[dependencies]
lash-runtime         = "=0.1.0-alpha.13"
lash-provider-openai = "=0.1.0-alpha.13"
anyhow               = "1"
tokio                = { version = "1", features = ["full"] }

One LashCore per app, cloned freely; one LashSession per chat or task; .run() for a single collected turn, .stream(&sink) for live events.

use lash::{provider::ProviderHandle, LashCore, ModelSpec, TurnInput};
use lash_provider_openai::{OPENROUTER_BASE_URL, OpenAiCompatibleProvider};

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let api_key = std::env::var("OPENROUTER_API_KEY")?;
    let provider = ProviderHandle::new(
        OpenAiCompatibleProvider::new(api_key, OPENROUTER_BASE_URL).into_components(),
    );

    let model = ModelSpec::from_token_limits(
        "anthropic/claude-sonnet-4.6", None, 200_000, None, None,
    )
    .map_err(anyhow::Error::msg)?;

    // one LashCore per app, cloned freely.
    let core = LashCore::standard()
        .provider(provider)
        .model(model)
        .build()?;

    // one session per chat / task; run one turn; read settled prose.
    let session = core.session("hello-1").open().await?;
    let result = session
        .turn(TurnInput::text("Say hi in one short sentence."))
        .run()
        .await?;

    println!("{}", result.assistant_message().unwrap_or_default());
    Ok(())
}

§ 04

the workspace

the workspace

lash is a Rust workspace layered from a pure state machine up to an app-facing facade. You depend on the facade and a provider; the rest is there when you need to reach under it.

lash-sansio
The pure turn machine — prompt model, messages, effects, responses, checkpoints, tool contracts, and canonical tool-call output. No Lashlang dependency.
lash-core
Async runtime internals: plugin host, protocol build input, providers, persistence, session graph, child-session orchestration, built-in tools, Lashlang host surface.
lash
The app-facing facade — runtime construction, sessions, turn streaming, and provider / mode / plugin wiring. Published as lash-runtime.
lash-protocol-*
standard and rlm protocol plugins — the two execution modes as installable units.
lashlang
The RLM execution language: parser, sandboxed VM, and projection. The surface the model writes against.
lash-cli
A first-party terminal frontend built on the library — the agent you can run, not just embed.
