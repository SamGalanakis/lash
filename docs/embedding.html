lash-embed guide

embedding/api

lash-embed is the app-facing layer for building Lash into a web service, desktop app, benchmark harness, or product workflow. Your app owns storage, auth, HTTP, frontend protocols, and product state; Lash owns model turns, modes, tools, plugins, semantic stream events, usage, and terminal outcomes.

Core Shape

Build one shared LashCore for app-wide configuration, then open one LashSession per app conversation or task.

Embed host boundary
use std::sync::Arc;

use lash::ToolProvider;
use lash_embed::{Input, LashCore, ModeId, ModePreset};

let core = LashCore::builder()
    .install_mode(ModePreset::standard())
    .install_mode(ModePreset::rlm())
    .default_mode(ModeId::rlm())
    .provider(provider)
    .model("anthropic/claude-sonnet-4.6", None)
    .max_context_tokens(200_000)
    .tools(Arc::new(AppTools) as Arc<dyn ToolProvider>)
    .store_factory(store_factory)
    .build()?;

let session = core.session("chat-123").rlm().open().await?;
let result = session.turn(Input::text("Use the app tools.")).run().await?;
let assistant_text: String = result.activities.iter().filter_map(|activity| match &activity.event {
    TurnEvent::AssistantProseDelta { text } => Some(text.as_str()),
    _ => None,
}).collect();
println!("{assistant_text}");
LashCore
Cloneable shared configuration: provider, model, installed modes, tool providers, plugin factories, store factory, attachment store, tracing, residency, sanitizer, and termination policy.
ModePreset
Installs execution modes. Use ModePreset::standard() for native provider tool calls, ModePreset::rlm() for Lashlang-driven RLM turns, or install both and choose the default with default_mode.
LashSession
One app conversation or task. Sessions wrap a parked/resumable runtime, can use a per-session store, and expose turn(Input), run(Input), read_view(), and rlm_project(...).
TurnBuilder
Per-turn configuration: cancellation, mode options, typed plugin input, and RLM-projected bindings. Call .stream(&sink) for live events or .run() for a collected ordered activity log.

Run A Turn

TurnBuilder::run() is the smallest path. It returns a CollectedTurnResult with the ordered Vec<TurnActivity> emitted during the turn.

Collected result

let collected = session
    .turn(Input::text("Summarize this task."))
    .run()
    .await?;

let visible_answer: String = collected.activities.iter().filter_map(|activity| match &activity.event {
    TurnEvent::AssistantProseDelta { text } => Some(text.as_str()),
    _ => None,
}).collect();
let usage = collected.usage;
let outcome = collected.outcome;

Live stream

let ui_sink = Arc::new(AppEvents::new(tx));

let turn = session
    .turn(Input::text(user_text))
    .stream(ui_sink.as_ref())
    .await?;

persist(app_turn_state.assistant_text(), turn.usage)?;

Apps own their projection policy. Fold TurnActivity directly when persisting assistant prose, terminal values, tool summaries, or timelines.

Semantic Events

The app stream is additive. Assistant prose arrives as deltas, tool and code activity arrive as structured events, and terminal values authored by submit or tool controls arrive as TerminalOutput.

TurnActivity sink
use async_trait::async_trait;
use lash_embed::{TurnActivity, TurnActivitySink, TurnEvent};

struct AppEvents;

#[async_trait]
impl TurnActivitySink for AppEvents {
    async fn emit(&self, activity: TurnActivity) {
        match activity.event {
            TurnEvent::AssistantProseDelta { text } => {
                append_live_text(text).await;
            }
            TurnEvent::ToolCallStarted { name, args, .. } => {
                show_tool_started(name, args).await;
            }
            TurnEvent::ToolCallCompleted { name, result, success, .. } => {
                show_tool_result(name, result.raw, success).await;
            }
            TurnEvent::CodeBlockStarted { language, code } => {
                show_code_block(language, code).await;
            }
            TurnEvent::TerminalOutput { source, value } => {
                append_live_text(render_terminal_value(&value)).await;
                record_terminal_source(source).await;
            }
            TurnEvent::Usage { usage, cumulative, .. } => {
                update_usage(usage, cumulative).await;
            }
            TurnEvent::ReasoningDelta { .. }
            | TurnEvent::CodeBlockCompleted { .. }
            | TurnEvent::Error { .. }
            | _ => {}
        }
    }
}

fn render_terminal_value(value: &serde_json::Value) -> String {
    match value {
        serde_json::Value::Null => String::new(),
        serde_json::Value::String(text) => text.clone(),
        other => serde_json::to_string_pretty(other).unwrap_or_else(|_| other.to_string()),
    }
}
Do not synthesize a second final answer.

TurnEvent::TerminalOutput means “a new terminal value was authored by a control path.” It is not emitted for a normal assistant prose finish because that prose already streamed as AssistantProseDelta.

RLM And Submit

RLM turns can require submit or allow direct prose. With SubmitRequired, a submit statement validates against the configured schema when present, finishes the turn as TurnFinish::Value, and emits a TerminalOutput semantic event. With ProseOrSubmit, a no-code prose answer finishes as TurnFinish::AssistantMessage and emits no TerminalOutput.

Per-turn RLM options

use lash::{ExecutionMode, ModeTurnOptions};
use lash_rlm_types::RlmTermination;

let options = ModeTurnOptions::typed(
    ExecutionMode::new("rlm"),
    RlmTermination::SubmitRequired {
        schema: None,
    },
)?;

let prose_options = ModeTurnOptions::typed(
    ExecutionMode::new("rlm"),
    RlmTermination::ProseOrSubmit,
)?;

let result = session
    .turn(Input::text("Move on the board."))
    .mode_turn_options(options)
    .stream(&sink)
    .await?;

Outcome shape

match result.outcome {
    lash::TurnOutcome::Finished(
        lash::TurnFinish::Value {
            source: lash::TerminalOutputSource::RlmSubmit,
            value,
        }
    ) => {
        // Same value already arrived as TurnEvent::TerminalOutput.
        persist_typed_value(value)?;
    }
    lash::TurnOutcome::Finished(
        lash::TurnFinish::AssistantMessage { text }
    ) => persist_text(text)?,
    other => handle_other_outcome(other)?,
}

Prompt Templates And Slots

lash-embed exposes the same prompt model as the runtime: one effective template chooses the layout, while slot contributions supply app, session, and turn-specific content. Core, session, and turn layers inherit by default; a lower layer can replace or clear one slot without rebuilding the whole template.

Template layout is separate from slot content
use lash_embed::{
    PromptBuiltin, PromptContribution, PromptSlot, PromptTemplate,
    PromptTemplateEntry, PromptTemplateSection,
};

let template = PromptTemplate::new(vec![
    PromptTemplateSection::untitled(vec![
        PromptTemplateEntry::builtin(PromptBuiltin::MainAgentIntro),
        PromptTemplateEntry::slot(PromptSlot::Intro),
    ]),
    PromptTemplateSection::titled(
        "Guidance",
        vec![PromptTemplateEntry::slot(PromptSlot::Guidance)],
    ),
]);

let core = lash_embed::LashCore::standard()
    .provider(provider)
    .model("gpt-5.4", None)
    .max_context_tokens(200_000)
    .prompt_template(template)
    .prompt_contribution(PromptContribution::guidance(
        "App",
        "Answer as the host application assistant.",
    ))
    .build()?;

let session = core
    .session("customer-42")
    .replace_prompt_slot(
        PromptSlot::Guidance,
        [PromptContribution::guidance(
            "Tenant",
            "Use the tenant's support policy.",
        )],
    )
    .open()
    .await?;

let result = session
    .turn(Input::text("Draft the response."))
    .prompt_contribution(PromptContribution::guidance(
        "Turn",
        "Keep this reply under 120 words.",
    ))
    .run()
    .await?;

Typed Plugin Input

Embed plugins let an app provide strongly typed session configuration and per-turn input. The plugin can contribute prompt context and tools, and tools read the same typed turn input through ToolExecutionContext.

Plugin and tool pattern
#[derive(Clone, Debug)]
struct BoardPlugin;

#[derive(Clone, Debug)]
struct BoardConfig;

#[derive(Clone, Debug)]
struct BoardTurnInput {
    board: BoardState,
}

impl lash_embed::EmbedPlugin for BoardPlugin {
    const ID: &'static str = "board";
    type SessionConfig = BoardConfig;
    type TurnInput = BoardTurnInput;

    fn factory(_: &Self::SessionConfig) -> Arc<dyn lash::PluginFactory> {
        Arc::new(BoardPluginFactory)
    }

    fn requires_turn_input(_: &Self::SessionConfig) -> bool {
        true
    }
}

impl lash::SessionPlugin for BoardSessionPlugin {
    fn id(&self) -> &'static str { BoardPlugin::ID }

    fn register(&self, reg: &mut lash::PluginRegistrar) -> Result<(), lash::PluginError> {
        reg.prompt().contribute(Arc::new(|ctx| {
            Box::pin(async move {
                let Some(input) = ctx.turn_context.plugin_input::<BoardTurnInput>(BoardPlugin::ID) else {
                    return Ok(Vec::new());
                };
                Ok(vec![lash::PromptContribution::environment(
                    "Board",
                    format_board(&input.board),
                )])
            })
        }));
        reg.tools().provider(Arc::new(BoardTools))
    }
}

#[async_trait::async_trait]
impl lash::ToolProvider for BoardTools {
    fn definitions(&self) -> Vec<lash::ToolDefinition> {
        vec![/* read_board, play_move, app tools */]
    }

    async fn execute_with_context(
        &self,
        name: &str,
        args: &serde_json::Value,
        context: &lash::ToolExecutionContext,
    ) -> lash::ToolResult {
        let Some(input) = context.turn_context.plugin_input::<BoardTurnInput>(BoardPlugin::ID) else {
            return lash::ToolResult::err_fmt("missing board input");
        };
        run_board_tool(name, args, &input.board)
    }
}

Human input follows the same rule: expose it as a host-owned tool that waits inside its tool implementation and returns a normal tool result. Embed streams then see ordinary tool start/completion events; there is no separate runtime prompt event to handle.

Register the plugin on the core, activate it on the session, and provide turn input before each run:

let core = LashCore::rlm()
    .register_plugin::<BoardPlugin>()
    .provider(provider)
    .model(model, None)
    .build()?;

let session = core
    .session(chat_id)
    .rlm()
    .use_plugin::<BoardPlugin>(BoardConfig)?
    .open()
    .await?;

let result = session
    .turn(Input::text("Play one move."))
    .with_plugin_input::<BoardPlugin>(BoardTurnInput { board })
    .stream(&sink)
    .await?;

App Storage

Keep product storage separate from runtime storage. Your app database can store chats, users, messages, and UI state; Lash session persistence stores the runtime graph and checkpoints through a SessionStoreFactory.

App state

Own chat tables, account ids, frontend board state, request auth, and transport formats. The example app stores chat messages in SQLite and streams newline-delimited JSON to the browser.

Runtime state

Provide .store_factory(...) on LashCoreBuilder or .store(...) on SessionBuilder when sessions need durable runtime state across process restarts.
