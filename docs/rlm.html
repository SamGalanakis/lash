rlm/protocol

How RLM mode builds model requests, projects history and variables into Lashlang, and turns model-authored code into one settled Lash turn.

What RLM Owns

RLM is a core execution mode, not a CLI feature. Hosts install it with ModePreset::rlm() and run normal LashSession turns; the protocol plugin changes how the model acts.

One action is a program

The model writes a single fenced lashlang block. Lash executes that block in a persistent VM, records its print output and terminal value, then either asks the model for another block or commits the turn.

Effects stay host-owned

Lashlang has no direct filesystem, process, network, or provider access. Every operation crosses the linked LashlangSurface and the runtime effect controller.

let core = LashCore::builder()
    .install_mode(ModePreset::rlm())
    .default_mode(ModeId::rlm())
    .plugins(lash::plugins::runtime_plugin_stack())
    .build()?;

Provider Request Shape

RLM sends the provider a normal chat-style request, but with no native provider tools. Tool and host work happens only after the model emits Lashlang.

One RLM iteration
LlmRequest {
  messages: [
    System(rendered system prompt),
    ...chronological history messages,
    User(current iteration tail),
  ],
  tools: [],
  tool_choice: None,
}
System prompt
Rendered from the active prompt template. Its execution-instructions slot contains the RLM Lashlang instructions for the effective host surface, plus prompt contributions such as showcased tools and variable sections.
History messages
Prior user messages, tool calls, protocol events, and RLM steps are projected as separate model messages. User messages and tool calls render as user-role context; prior RLM steps and protocol events render as assistant-role context.
Current iteration tail
A volatile user-role message containing the iteration number, active turn causes, finalization rule, optional required-output schema, optional final-answer format, and optional context-budget guidance.

System Prompt Construction

The system prompt is the stable instruction prefix. It tells the model how to write Lashlang and which host surface is callable; it does not inline the full runtime state.

The variable sections are advertisements, not the data transport. Execution receives real projected bindings separately, immediately before Lash runs the fenced code.

### Bound Variables

These variables are already bound in lashlang. Access them directly in fenced `lashlang` code; do not recreate them manually.
Type and size hints help you plan before inspecting a value.

Available variables:
- `history`: `list<HistoryItem>`, Readonly: true, projected binding, 3 entries
- `task`: `Task`, Readonly: true, projected binding, keys=4

### Host Projected Variables

Host-projected variables:
- `board`: projected host value, Readonly: true, projected binding

History Projection

RLM keeps chronological context readable while avoiding giant prompt blobs. Long fields are head/tail truncated in the model message and paired with a history[...] handle for full lookup from Lashlang.

--- history[0] · user message · 18 chars ---

Find the failing test.

--- history[1] · rlm step · protocol_iteration 1 ---

Reasoning (31 chars):
I should inspect the test output.

Code:
```lashlang
result = await workspace.default.shell({ command: "cargo test" })?
print { status: result.status, tail: slice(result.stderr, -1200, null) }
```

history[1].output[0] (1250 chars, full: history[1].output[0]):
{ "status": 101, "tail": "..." }

The same history name is available inside Lashlang as a read-only projected binding. If a rendered preview says full: history[1].output[0], the model can recover the untruncated value by reading or printing that path.

Messages
Rendered as history[N].content, with attachments exposed under history[N].attachments[M].
Tool calls
Rendered with arguments and result previews; full values are reachable as history[N].args and history[N].result when truncated.
RLM steps
Rendered with reasoning, the exact Lashlang code, print outputs, images, errors, and final submitted output. Print outputs are addressable as history[N].output[M].

Variables And State

There are three state channels. Keeping them distinct is the key to understanding RLM prompts.

VM globals

Variables assigned by fenced blocks persist in the active AgentFrame's RLM execution state. A fresh AgentFrame starts a clean namespace unless values are passed explicitly through seed.

Projected bindings

history plus host-injected values are read-only names. Session bindings and per-turn bindings are merged before execution; duplicate names are rejected.

Prompt hints

The prompt lists names, types, and size hints. It does not paste every bound value into the system prompt.

```lashlang
print {
  history_entries: len(history),
  board_turn: board.turn,
  board_cells: len(board.cells)
}
```

Projected names cannot be reassigned. A write like history = [] or board = {} fails with a read-only projected binding error; copy the fields you need into a new local variable instead.

Execution Loop

Each RLM iteration has one model response and at most one executed Lashlang block. Anything after the first closed ```lashlang fence is ignored by the driver.

1. prompt

The runtime renders the system prompt, projects chronological history, appends the current iteration tail, and sends the provider request.

2. code

The model emits prose plus one closed lashlang block. Stream masking keeps raw code out of normal assistant prose surfaces.

3. execute

The executor resolves projected bindings, links the code against the effective LashlangSurface, and runs it in the current VM state.

4. observe

print output, operation results, diagnostics, and images become the next iteration's history context.

5. finish

submit finishes the turn as TurnFinish::SubmittedValue or rendered prose. control.continue_as(...) ends the current trajectory and opens a fresh AgentFrame with explicit seed state.
