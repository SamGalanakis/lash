Architecture comparison · Recursive Language Models

How lash and DSPy put REPL history into the prompt

Both run the same loop: the model emits one fenced code block, the host executes it, the model sees the result and iterates until it submits. They diverge on how the trajectory re-enters the prompt. lash maintains an incremental, cache-fenced multi-message conversation with a live type-hinted variable namespace. DSPy statelessly re-renders the whole trajectory as a single text field on every iteration.

1 The shared model

What both implementations agree on

These are the same idea, executed twice. Before the differences, it's worth pinning down the substantial common ground — the RLM contract is identical down to the default truncation constant.

Identical

The REPL loop

Model writes code → host executes → model reads output → iterates → submits. Persistent state survives across steps.

Identical

One fenced block per turn

Exactly one code block per response. lash recognises only ```lashlang; DSPy accepts ```python (and bare fences).

Identical

Inspect vs. finish

print to inspect a value and keep going; submit / SUBMIT() to return the final answer and end the turn.

Identical

Truncation strategy

Head + tail truncation, middle elided. Same default: max_output_chars = 10_000 in both codebases.

Identical

Reasoning + code outputs

Each step the model produces a reasoning field and a code field; the host records reasoning, code, and output per step.

Identical

Values stay out of band

Both push the "don't hand-copy large values; keep them in variables and print slices" discipline into the instructions.

2 Prompt assembly mechanism

The biggest difference: conversation vs. re-render

This is the structural fork everything else follows from. lash treats the turn as a growing conversation it appends to and caches; DSPy treats every iteration as a fresh, stateless prediction whose entire prompt is rebuilt from the trajectory object.

lash · incremental, cache-fenced

Builds a real multi-message conversation. Each past RLM step becomes an Assistant-role message; user / tool entries become User messages.

  • System preamble built once per turn driver.rs:49 build_rlm_preamble
  • The last history message is marked with a cache_breakpoint history.rs:185 mark_last_history_text_cache_breakpoint
  • Only the volatile === CURRENT ITERATION === tail (iteration no., turn events, finalization, required-output schema, context budget) is re-sent uncached history.rs:97 append_current_iteration_message
  • The provider reuses the cached KV prefix turn over turn
DSPy · stateless re-render

Every iteration calls a fresh dspy.Predict with the full trajectory passed as input fields.

  • self.generate_action(variables_info=…, repl_history=history, iteration=…) each loop rlm.py:600
  • The ChatAdapter re-renders the whole prompt, wrapping each field in [[ ## field_name ## ]] delimiters chat_adapter.py
  • History is one big InputField of type REPLHistory that serializes to a single string repl_types.py:139
  • No cross-turn caching contract — the prompt is rebuilt from scratch
The consequence in one line

lash spends design complexity to make the prompt append-only and cacheable; DSPy spends tokens to keep the prompt portable and adapter-agnostic. Sections 3–6 are downstream of this choice.

4 Variable / globals surfacing

Live type-map vs. fixed input previews

The sharpest contrast. lash shows a live, full namespace as types and size hints (values by reference). DSPy shows fixed previews of the signature inputs only — variables the model creates mid-run are visible only through the transcript.

5 Recursion & loop termination

Sub-LLM calls vs. spawned agents; iteration cap vs. token budget

The "recursive" in RLM means different machinery in each. DSPy's defining primitive is the in-REPL sub-LLM call; lash routes decomposition through its general agent/tool surface. Termination differs too.

lash

Recursion: no dedicated sub-LLM primitive in the prompt. Decomposition goes through agents.spawn and the general tool / process surface.

Termination: bound by a token budget, not an iteration count. Frame-switch nudges via control.continue_as escalate at 60 % / 90 % / 100 %. rlm_support.rs format_budget_suffix · default 100k tokens

DSPy

Recursion: the defining feature is llm_query(prompt) / llm_query_batched(prompts) — recursive sub-LLM calls capped by max_llm_calls (default 50). rlm.py:227

Termination: bound by max_iterations (default 20). On exhaustion a separate extract Predict forces a final output. rlm.py:466 _extract_fallback

6 Implications

Tradeoffs, not a winner

Each choice buys something and costs something. The table names both sides per dimension.

Dimension lash DSPy
Caching cost Reuses the provider KV cache across turns; only the volatile tail is uncached.Pays complexity: cache-breakpoint placement, message roles, volatile/stable split. Pays a full prompt re-render every iteration.Buys simplicity: no caching contract to maintain.
Context-rot exposure Variables surfaced as types/sizes keep raw bytes out of the prompt until printed. Input previews are capped, but the full trajectory string grows in-band each step.
Scaling with long trajectories Stable prefix stays cached; appended steps are the marginal cost. Whole repl_history re-serialized and re-sent every loop — cost grows with both length and iteration count.
Variable awareness Stateful & complete: live type-map of every current global, refreshed each turn. Fixed & partial: input previews only; derived vars live in the transcript.
Where errors surface Dedicated Error: field per step. Folded into the output string ([Error] / [Type Error]).
Truncation re-fetch Addressable full: history[i].output[j] handle — model can pull the full value back. (K characters omitted) with no handle to recover the elided middle.
Recursion primitive General agents.spawn + tool/process surface. Purpose-built llm_query / llm_query_batched.
Loop bound Token budget with 60/90/100 % frame-switch nudges. max_iterations count + extract fallback.
Portability Provider-cache-aware multi-message; richer but coupled to cache semantics. Adapter-agnostic single-field; trivially portable across any chat backend.
Net

lash optimizes for long-running, cache-efficient turns with full state awareness — at the cost of a more intricate prompt-assembly contract. DSPy optimizes for portability and a clean stateless module boundary — at the cost of re-paying the full prompt every iteration and a thinner sense of the live namespace.
