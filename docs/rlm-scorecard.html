The honest call

It depends what you're building — and both nailed their own target.

On the dimensions themselves, lash is the stronger design: it wins caching, context-rot, scaling, variable awareness, error surfacing, truncation re-fetch, and (just) termination. Those are the dimensions that decide whether a long-running, expensive, stateful agent turn stays cheap and coherent. That is lash's actual job, so for lash, lash's approach is the right one — clearly.

DSPy wins portability, implementation simplicity, and recursion ergonomics. Those are the dimensions that decide whether a module drops cleanly into any pipeline on any chat backend — DSPy's actual job. Its losses on caching and scaling don't bind that goal: a stateless single-field prompt is a feature when portability is the point.

If forced to crown one design in the abstract: lash, on points (7.7 vs 6.4) — it carries more correctness and efficiency machinery. But the fairer statement is that each is near-optimal for its own constraints, and the two scorecards barely overlap in what they reward.

Wholehog score · lash approach vs. the ideal end-state

Not vs. DSPy — vs. the clean endpoint

This asks a different question: how close is lash's history/variable rendering to the simplest correct final design, with no dead paths, drift, or unevidenced contracts? Higher is closer to ideal.

88/ 100

Solid, near-ideal — ship it, then close two gaps

scope: rendering RLM trajectory + variables into the model prompt (lash-protocol-rlm), design boundary

Why not lower: the endpoint is singular and clear — one model (cache-fenced multi-message + a memoized type-hinted bound-variables contribution), the old duplicate tool-call history section was already deleted, and cache boundaries plus variable surfacing are genuine end-state work, not milestone shims. Why not higher: a #[cfg(test)] single-string render path appears to re-implement the production multi-message renderer (drift risk), and the advertised full: history[i].output[j] re-fetch contract isn't proven to resolve. Driven by the Evidence and Cutover-completeness lenses.

first score for this target · establishes the baseline

Good — already at the ideal

  • One coherent model. History as messages + a bound-variables contribution; no competing prod rendering paths. history.rs · rlm_support.rs
  • Cutover evidence in-tree. The old per-tool-call history section was deleted because print result + the code field already covered it. history.rs:666
  • End-state caching. Cache-breakpoint on the last history message; volatile iteration tail kept uncached. history.rs:185, :97
  • Values by reference. Type + size hints + generated schemas, memoized by Arc identity + history_len — not raw dumps.

Needs improving — the gap to 100

  • Test-only render path. render_history_prompt / render_history_item under #[cfg(test)] look like a second renderer; if they don't exercise append_repl_step, tests can pass while prod drifts. history.rs:205+
  • Unevidenced re-fetch contract. The prompt tells the model it can recover a truncated value via history[i].output[j] — needs a test proving that path actually resolves the full value.
  • Vestigial field. ExecResponse.output is essentially always empty (observations ride a separate vec); the "raw stdout" branch is a near-dead path in the prompt builder. driver.rs:215
  • Out of scope but adjacent: the CLI drops CodeBlockCompleted — display only, not prompt rendering, so it doesn't move this score.
  1. +5Collapse the test-only renderer onto the prod path. Make history tests assert against append_repl_step output directly, delete any parallel single-string formatter. Removes drift risk — the largest wholehog deduction. history.rs:205+
  2. +4Add a re-fetch integration test. Truncate an output, then prove the model can print history[i].output[j] and get the full value back. Turns an advertised contract into an evidenced one.
  3. +2Retire the vestigial output string or fold raw stdout into observations, so the prompt builder has one output channel, not two. Simplicity + boundary cleanup. driver.rs:215
  4. +1Document the prompt contract (message roles, cache fence, bound-variables semantics) next to the renderer so the design intent is legible. Evidence/legibility polish.
Practically done at ~92 once steps 1–2 land — beyond that it's diminishing returns unless a concrete token-budget or provider-cache SLA says otherwise.
